| 维度 | 用户的提示词（✅ 正面教材） | 反面教材（❌ 反面教材） | 为什么这是正确的？ |

| **🔲 边界值处理** | 考虑了前端/后端/全栈三种情况，明确了并行调用策略 | 只考虑正常情况，边缘场景完全不提 | 🎯 **架构师思维**：好的命令设计要覆盖所有组合情况，不能让用户遇到边界情况时不知所措 |

| **⚡ 性能影响** | 要求前后端都存在时"同时调用"（并行），而不是串行 | "先调前端，成功了再调后端" | 🎯 **架构师思维**：有性能意识，知道并行可以提升效率，不是只关心功能对不对 |

### 1.3 用户的关键决策点及其正确性分析

| 决策点 | 用户的选择 | 正确性分析 | 架构师思维 |
|---------|-----------|-----------|------------|
| **决策 1：从 Git 驱动改为对话上下文驱动** | 放弃基于 Git diff 的变更检测，改为基于当前对话内容分析 | ✅ **完全正确**。Git 是事后的，对话是实时的。真正的 AI 协同应该发生在对话过程中，而不是等提交代码后 | 🔍 **体验驱动思维**：用户在对话过程中随时想保存成果，Git 提交的时机和协同完成的时机不一定重合。技术应该适应人的工作流，而不是让用户适应技术 |
| **决策 2：智能识别类型并自动调用对应命令** | 不是让用户手动选调用哪个命令，而是 AI 自动分析对话内容，智能判断 | ✅ **完全正确**。用户只需要关心"我要保存"，不需要关心"保存需要调用哪几个子命令" | 🎯 **开闭原则**：对扩展开放，对修改关闭。新增命令类型时，只需要修改识别规则，用户的使用方式不变 |
| **决策 3：前后端都有时并行调用** | 明确要求"同时调用"，而不是串行 | ✅ **完全正确**。前后端文档生成是独立的，没有依赖关系，完全可以并行 | ⚡ **性能优化思维**：识别出可并行的任务，这是架构师的核心能力之一。不要把可以并行的任务写成串行 |
| **决策 4：强制调用 teaching-summary** | 从"可选"改为"必须" | ✅ **完全正确**。教学沉淀是组织级别的核心价值，不能让用户"忘记"做。好的工具应该把最佳实践固化到流程中 | 🛡️ **防护性设计**：把重要的事情从"用户的自觉"变成"工具的强制"，这是从"个人优秀"到"团队优秀"的关键一步 |
| **决策 5：不破坏现有命令的独立性** | frontend-module-doc 和 backend-module-doc 保持独立，可以单独使用，也可以被 save-ai 调用 | ✅ **完全正确**。组合式设计，每个命令既是独立的工具，也是更大工作流的一部分 | 🧩 **组合优于继承**：Unix 哲学——每个工具只做一件事，并且做好。复杂功能通过管道/组合实现，而不是写大而全的工具 |

## 3. 方案对比：Git 驱动 vs 对话上下文驱动

### 修复方案对比（6 维度量化打分制）

> 📋 **场景类型**：复杂业务场景（AI 协同工具设计）
> 🔢 **权重配置**：功能 40%、边界 30%、安全 10%、性能 20%

| 评估维度 | 方案 A：继续优化 Git 驱动方案 | 方案 B：改为对话上下文驱动 |
|----------|-----------------------------|---------------------------|
| 🧩 功能完整性 | ⭐⭐ 只能在 Git 提交后保存，对话中的中间成果无法保存 | ⭐⭐⭐⭐⭐ 对话中随时可以保存，支持中间成果和实验性代码 |
| 🔲 边界值处理 | ⭐⭐ Git 没有变更时无法工作，多模块变更时很难准确识别 | ⭐⭐⭐⭐⭐ 对话内容是最准确的上下文，不存在"没有变更"的问题 |
| 🔒 安全考虑 | ⭐⭐⭐⭐⭐ 不涉及安全问题 | ⭐⭐⭐⭐⭐ 不涉及安全问题，都是只读操作 |
| ⚡ 性能影响 | ⭐⭐⭐ 需要执行 Git 命令，有 IO 开销，文件多时慢 | ⭐⭐⭐⭐⭐ 纯内存分析，不需要 IO，速度快 |
| 🔧 可维护性 | ⭐⭐ Git 检测逻辑会越来越复杂，边界情况越来越多 | ⭐⭐⭐⭐⭐ 分析对话是 AI 的强项，逻辑简单清晰 |
| 💰 开发成本 | ⭐⭐⭐⭐⭐ 只需要小修小补，1 小时搞定 | ⭐⭐⭐⭐ 完整重写，需要 2-3 小时 |
| **加权总分** | **10.8 / 20** | **18.5 / 20** |
| **🚫 否决项检查** | ✅ 通过 | ✅ 通过 |
| **用户选择** | ❌ 放弃 | ✅ 选中 |

> 💡 **方案选择理由**：
> - 方案 B 在功能完整性和边界值处理上有碾压性优势，这两个维度权重占了 70%
> - 开发成本虽然略高，但这是一次性投入，长期收益巨大
> - 最关键的是：方案 A 的方向是错的，越优化离用户的真实需求越远
> - 🎯 **架构师思维**：当方向错了的时候，停下来就是进步。不要在错误的方向上继续优化，及时止损才是正确的选择

## 6. 教学总结

### 6.1 学到的核心原则

1. **方向比速度更重要**
   - 当用户说"方向不对"的时候，不要继续在错误的方向上优化，停下来重新思考
   - Git 驱动 vs 对话驱动，是根本方向的差异，不是细节优化的问题

2. **技术应该适应人的工作流，而不是反过来**
   - 用户在对话过程中随时想保存成果，Git 提交的时机和这个时机不一定重合
   - 好的工具应该配合用户的工作方式，而不是要求用户改变习惯来适应工具

3. **组合优于修改**
   - 不修改现有的 frontend-module-doc 和 backend-module-doc，而是通过 save-ai 把它们组合起来
   - 每个命令既是独立的工具，也是更大工作流的一部分，这是 Unix 哲学的精髓

4. **把最佳实践固化为强制流程**
   - 教学总结是高价值的，但如果让用户"记得调用"，大概率会忘记
   - 好的工具应该把重要的事情从"用户的自觉"变成"工具的强制"，保证组织级别的价值积累

5. **识别可并行的任务，不要串行**
   - 前后端文档生成是独立的，没有依赖关系，完全可以并行
   - 架构师的核心能力之一就是识别出哪些任务可以并行，这能带来巨大的性能提升

### 6.2 常见踩坑点总结

1. **技术导向 vs 用户导向**
   - ❌ 踩坑：从技术的便利性出发（Git diff 很好实现），而不是从用户的真实需求出发
   - ✅ 避免：先问用户"你在什么场景下想用这个功能？"，再想技术怎么实现

2. **把 AI 工具做成普通命令行工具**
   - ❌ 踩坑：让用户手动传 `--front` / `--back` 参数，AI 明明知道用户在做什么
   - ✅ 避免：AI 工具的核心价值就是智能识别，把识别的工作留给 AI，把决策的工作留给用户

3. **可选的重要功能 = 不会被执行**
   - ❌ 踩坑：把教学总结设为可选，结果大部分时候都不会被调用
   - ✅ 避免：重要的、有长期价值的功能，一定要设为强制执行

4. **默认串行，忘记并行**
   - ❌ 踩坑：习惯性地按顺序写 A 然后 B 然后 C，没有思考 A 和 B 是否可以同时执行
   - ✅ 避免：设计任何流程时，先问自己：这几个步骤有依赖关系吗？没有的话能不能并行？

### 6.3 问题复盘清单（标准化流程）
