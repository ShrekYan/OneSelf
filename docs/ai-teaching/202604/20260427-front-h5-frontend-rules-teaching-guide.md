## 💬 提示词优化复盘

### ✅ 本次哪些提示词写得好？好在哪里？

1. **"先进入计划模式，给我三种不同的方案（评估方案的优劣），让我审查，我确认后再执行"**
   - 好处：先规划后执行，避免盲目开发，确保方案符合预期
   - 可复用性：所有涉及规范修改、架构调整的任务都应该先规划

2. **"types.ts 不需要定义 API 响应类型（DTO），因为 apps/web/src/types 这个文件夹已定义"**
   - 好处：用户及时澄清边界条件，避免生成错误的规范
   - 启示：涉及类型系统时，一定要先确认全局与局部的职责边界

### ⚠️ 哪些提示词可以改进？怎么改会更好？

原始需求相对简洁，可以更结构化：

```
# 改进前（原始）
背景：
我让AI协助进行性能检查时发现了一个问题。

可参考：apps/web/src/pages/Login

需求功能：新增/修改规则：
- 生成的页面结构添加types，专门定义页面的类型定义。
- 生成的页面结构添加hooks，当前业务逻辑复杂时拆分hooks
- handle.ts方法存放纯函数，严禁出现副作用

完成标准：
- 先进入计划模式，给我三种不同的方案（评估方案的优劣），让我审查，我确认后再执行
```

```
# 改进后
背景：
在进行前端性能审查时，发现现有 h5-frontend-developer skill 的页面生成规则存在三个问题：
1. types.ts 文件为可选，导致页面类型定义混乱
2. 缺少 hooks 拆分的明确标准，业务逻辑复杂时不知道何时拆分
3. handle.ts 纯函数约束不够严格，经常混入副作用代码

参考基准：apps/web/src/pages/Login 目录（已实现完整结构）

需求功能：
请优化 h5-frontend-developer skill 的规则文件：
- 将 types.ts 从可选改为必选，明确页面级类型与全局 DTO 的边界
- 新增 hooks 拆分阈值标准和规范文档
- 强化 handle.ts 的纯函数约束，添加检测清单和反例

完成标准：
1. 先进入计划模式，提供三种不同强度的方案（渐进式/独立式/完整重构）并评估优劣
2. 我选定方案后再执行修改
3. 所有修改必须与现有 Login 页面结构保持兼容
```

### 💡 下次遇到类似任务，最优的提问方式是什么？

对于规范修改类任务，建议包含以下要素：

1. **问题背景**：为什么要改？遇到了什么具体问题？
2. **参考基准**：有没有已实现的正确示例可以参考？
3. **边界条件**：有哪些约束？比如全局与局部的分工？
4. **方案要求**：需要提供几种方案？各自侧重什么？
5. **验收标准**：修改后如何验证正确性？

## 🤔 关键决策回顾

### 1. 方案强度选择：渐进式强化 vs 独立文件重构 vs 完整架构重构

| 方案 | 优点 | 缺点 | 决策依据 |
|------|------|------|---------|
| 渐进式强化 | 改动最小、向后兼容、快速落地 | 结构性提升有限 | ✅ 现有规范已较完善，Login 页面已验证结构，不需要大改 |
| 独立文件重构 | 模块化强、职责清晰 | 文件数量增加多、维护成本高 | ❌ 过度设计，3个规则点不值得拆成7-8个文件 |
| 完整架构重构 | 系统化、可自动化检测 | 改动量大、过度设计 | ❌ 杀鸡用牛刀，现有问题不需要这么重的方案 |

**决策**：选择方案一（渐进式强化）
**依据**：现有规范已相当完善，问题只是三个局部点需要强化，不需要结构性重构。

### 2. 类型分层边界：页面 types.ts vs 全局 types 目录

**决策**：API DTO 统一放在 `apps/web/src/types/`，页面级 `types.ts` 只存放跨组件共享类型

**依据**：
- 避免类型重复定义
- 全局 DTO 是多个页面共享的，应该统一管理
- 页面级类型是页面内部组件共享的，应该放在页面目录下

**启示**：任何系统设计都要先明确"边界"，而不是急于定义细节。

### 3. hooks 拆分阈值：量化标准 vs 主观判断

**决策**：采用 4 个量化阈值（满足任一即建议拆分）

| 阈值 | 数值 | 理由 |
|------|------|------|
| 代码行数 | > 50 行 | 超过 50 行的函数可读性下降 |
| Hook 依赖数 | > 3 个 | 依赖过多说明职责复杂 |
| 复用性 | 多组件复用 | 复用的逻辑应该独立封装 |
| 业务完整性 | 完整流程 | 完整业务流程应该独立可测试 |

**依据**：主观的"复杂时拆分"难以执行，必须提供可量化的判断标准。

## 🐛 踩坑记录与教训

### 坑 1：Agent 规范文件引用链处理

**现象**：修改一个规则文件后，需要同步更新其他文件中对它的引用

**根因**：规范文件之间存在依赖关系，形成了引用链

**解决**：修改前先梳理引用关系图，确保所有引用都被更新

**教训**：
> 文档型修改也要像代码一样考虑"依赖管理"，不能只改目标文件而忽略了引用它的文件。

### 坑 2：规则的"可执行性" vs "描述性"

**现象**：最初的规范只有"应该怎么做"的描述，缺少"怎么做才对"的具体标准

**根因**：混淆了"原则"和"操作指南"的区别

**解决**：
- 原则：告诉开发者"应该"做什么
- 操作指南：告诉开发者"怎么判断"做对了没有
- 反例：告诉开发者"什么是错的"

**教训**：
> 好的规范 = 清晰的原则 + 可量化的检查清单 + 正反例对比。
> 只讲原则不讲方法的规范是无法落地的。

## ✅ 最佳实践提炼

### 1. 规范文件的三层结构

**模式**：任何开发规范都应该包含这三层

```
第一层：原则（Why）
  - 为什么要有这个规范？
  - 不遵守会有什么问题？

第二层：规则（What）
  - 具体要怎么做？
  - 满足什么条件？
  - ✅ 正确示例

第三层：校验（How）
  - 怎么判断做对了？
  - 检查清单
  - ❌ 反例对比
```

**适用场景**：所有团队规范、代码风格指南、最佳实践文档

### 2. 拆分阈值量化法

**模式**：对于"何时拆分"这类主观判断问题，提供量化阈值矩阵

```
满足任一条件即建议拆分：
  - 代码行数：超过 X 行
  - 依赖数量：超过 Y 个
  - 复用场景：被 Z 个地方调用
  - 业务完整性：包含完整流程
```

**适用场景**：组件拆分、Hook 拆分、函数拆分、文件拆分

### 3. 副作用白名单机制

**模式**：对于"什么算副作用"这类边界模糊的概念，提供明确的黑名单 + 例外白名单

```
❌ 禁止（黑名单）：
  - 网络请求
  - DOM 操作
  - React Hook
  - 状态修改

✅ 允许（白名单 - 例外）：
  - Dialog.confirm()
  - Toast.show()
  - console.log()
  - Date.now()
```

**适用场景**：纯函数定义、副作用边界、安全风险判断

## 🎨 React 组件设计复盘

本次虽然是规范修改，但涉及到页面结构的核心设计原则：

### ✅ 做得好的地方

1. **目录结构高度标准化**
   - 每个页面都有固定的 7+ 个文件/目录
   - 新开发者可以快速找到对应代码

2. **职责边界清晰**
   - handle.ts：纯函数逻辑
   - useStore.ts：状态管理
   - hooks/：依赖 Hook 的业务逻辑
   - components/：UI 组件

3. **纯函数优先原则**
   - 能抽成纯函数的逻辑尽量抽成纯函数
   - 副作用尽量推到外层（Store 或 Hook）

### 🎯 可以改进的地方

1. **规范之间的交叉引用还可以更完善**
   - 每个规则文件末尾应该列出"相关规范"
   - 形成规范之间的知识网络

2. **缺少自动化 lint 规则支持**
   - 目前都是文档规范，依赖人工 code review
   - 未来可以考虑写 eslint 插件自动检测

## 🧠 MobX 状态管理分析

本次规范强化了一个重要原则：

**状态修改必须统一收口到 useStore.ts**

| 操作 | 存放位置 | 理由 |
|------|---------|------|
| 修改 MobX 状态 | useStore.ts | 状态修改必须可追踪、可调试 |
| 不修改状态的业务逻辑 | handle.ts | 纯函数可测试、可复用 |
| 依赖 Hook 的业务逻辑 | hooks/ | React 规则约束 |

**启示**：
> 状态是系统中最复杂的部分，"状态修改权"必须严格管控，不能散落在代码各处。

## 📝 规范文件编写方法论

### 1. 反例驱动开发（Counter Example Driven Development）

写任何规范时，先想"最常见的错误写法是什么"，然后针对性禁止。

```typescript
// 先想反例
❌ export default { ... }

// 再写规则
✅ 必须使用具名导出

// 最后说明理由
方便 tree-shaking，统一导入风格
```

### 2. 决策文档化

每个规范背后都应该有"为什么这么定"的说明，而不是只有"要这么做"的要求。

**不好的规范**：
> handle.ts 中不能调用 API

**好的规范**：
> handle.ts 中不能调用 API，因为：
> 1. API 调用属于副作用，会破坏纯函数的可测试性
> 2. API 调用通常需要修改 Loading 状态，应该放在 Store 中
> 3. 纯函数逻辑可以被多个组件/Hook 复用

### 3. 提供决策流程图

对于复杂的"放在哪里"问题，提供简单的决策树：

```
这段逻辑应该放哪里？
    ↓
是否需要修改 MobX 状态？ → 是 → useStore.ts
    ↓ 否
是否是纯函数？ → 是 → handle.ts
    ↓ 否
是否依赖 React Hook？ → 是 → hooks/useXxx.ts
    ↓ 否
组件内 inline
```
