# MongoDB 在博客项目中的适用场景介绍

除了日志存储，MongoDB 在博客项目中还有很多非常适合的使用场景。本文分析哪些场景适合用 MongoDB，为什么适合。

---

## 适合 MongoDB 的典型场景

### 1. 用户行为数据 - 浏览历史、阅读记录

如果要做「我的浏览历史」功能，这是 MongoDB 的绝佳场景：
- **特点**：每个用户的浏览记录数据量变化大（从几条到几千条），写入非常频繁
- **为什么适合**：
  - MongoDB 单文档写入性能优于 MySQL
  - 不需要强事务，可用性优先级高于极致一致性
  - 查询用户自己的浏览历史，按 userId 索引一次就能返回
  - 数据量大了之后更容易分片扩展

**数据模型示例**：
```javascript
{
  userId: "user-uuid-xxx",
  articleId: "article-uuid-xxx",
  title: "文章标题",
  coverUrl: "封面图",
  viewedAt: ISODate("2026-04-19T10:00:00Z"),
  readProgress: 0.75 // 阅读进度
}
```

---

### 2. 评论系统（树形结构）

评论本身就是天然的树形结构，MongoDB 内嵌文档比 MySQL 更方便：

**为什么适合**：
- MySQL 需要自关联表，查询多层嵌套评论需要多次 JOIN 或者应用层递归
- MongoDB 可以直接内嵌回复，一次查询获取整棵评论树
- 评论结构可以灵活扩展（支持点赞、折叠、@用户、编辑历史等）

**数据模型示例**：
```javascript
{
  _id: ObjectId("..."),
  articleId: "uuid-xxx",
  content: "这是一篇评论",
  author: {
    userId: "...",
    nickname: "...",
    avatar: "..."
  },
  createdAt: ISODate(),
  likesCount: 0,
  // 子评论直接内嵌，一次查询搞定
  replies: [
    {
      content: "这是一条回复",
      author: { ... },
      createdAt: ISODate(),
      likesCount: 0
    }
  ]
}
```

---

### 3. 文章草稿 / 临时编辑

用户在编辑文章过程中的草稿：

**为什么适合**：
- 草稿结构可能随编辑器功能迭代频繁变化
- MongoDB schema 灵活，不需要提前严格定义，也不需要 ALTER TABLE
- 每个草稿就是一个文档，天然对应一篇文章的草稿
- 过期草稿清理很方便，直接按时间批量删除

**数据模型示例**：
```javascript
{
  articleId: "uuid-xxx", // 新建文章草稿可能还没有 id
  title: "草稿标题",
  blocks: [ ... ], // 内容块可以灵活变化结构
  lastModified: ISODate(),
  // 随时可以加新字段，不需要改表
  autosave: true,
  wordCount: 1234
}
```

---

### 4. 消息通知系统

站内信/通知功能：

**为什么适合**：
- 通知类型多种多样（点赞、评论、关注、系统通知...）
- 不同类型通知需要携带不同的数据字段
- MySQL 需要表拆分或者用 JSON 字段，MongoDB 天然支持灵活结构

**数据模型示例**：
```javascript
// 点赞通知 - 不同类型带不同数据
{
  userId: "target-user-id",
  type: "like",
  articleId: "...",
  likedBy: { id: "...", name: "...", avatar: "..." },
  read: false,
  createdAt: ISODate()
}

// 系统通知 - 可以有完全不同的字段
{
  userId: "user-id",
  type: "system",
  title: "系统公告",
  content: "我们更新了用户协议...",
  link: "/notice",
  read: false,
  createdAt: ISODate()
}

// 评论通知
{
  userId: "author-id",
  type: "comment",
  articleId: "...",
  commentId: "...",
  content: "用户...评论了你的文章",
  read: false,
  createdAt: ISODate()
}
```

---

### 5. 点赞 / 收藏关系

你现在 `ArticleLikes` 在 MySQL 里，但其实这个场景也很适合 MongoDB：

**为什么适合**：
- 写入频繁，每次点赞都要插入/删除
- 查询用户点赞列表，MongoDB 查询更快
- 不需要和其他表复杂关联，可以独立存在
- 数据量增长快，MongoDB 水平扩展更容易

**注意**：需要唯一约束，MongoDB 也支持唯一索引，可以保证不重复点赞。

**数据模型示例**：
```javascript
{
  articleId: "...",
  userId: "...",
  createdAt: ISODate()
}
// 唯一索引：{ articleId: 1, userId: 1 } 保证不重复
```

---

### 6. 访客统计 / UV PV 统计

文章访问量统计：

**为什么适合**：
- 写入极其频繁，每个页面访问都可能需要更新
- 是聚合统计场景，对一致性要求不是绝对强（允许少量误差）
- MongoDB 的原子更新 `$inc` 非常方便

**数据模型示例**：
```javascript
// 按天统计文档
{
  date: "2026-04-19",
  articleId: "article-xxx",
  pv: 1234,
  uv: 567,
  visitors: [ "user1-id", "user2-id", ... ] // 或者用单独集合
}
```

---

### 7. 操作审计日志

管理员操作记录、用户敏感操作记录：

**为什么适合**：
- 写入多，读取少（只有审计的时候才查询）
- 操作内容结构不固定，不同操作参数不同
- 数据会持续增长，MongoDB 更容易做数据分片归档
- 不需要和其他表关联

---

### 8. 用户标签 / 兴趣标签

如果要做个性化推荐：

**为什么适合**：
- 每个用户标签集合大小不固定
- 标签结构可以动态添加（随着推荐算法迭代）
- 适合用 MongoDB 存储用户画像

## 当前项目中按推荐优先级排序

| 场景 | 推荐度 | 为什么适合 | 当前是否已有 | 迁移收益 |
|------|--------|-----------|-------------|----------|
| **访问日志** | ⭐⭐⭐⭐⭐ | 写入量大，不需要事务，查询灵活 | 本地文件 | 方便查询统计，解决文件管理问题 |
| **文章内容块** | ⭐⭐⭐⭐⭐ | 文档型结构，灵活扩展，减少JOIN | MySQL | 提升查询性能，方便扩展块类型 |
| **用户浏览历史** | ⭐⭐⭐⭐ | 新增功能，写入频繁 | 无 | 从零开始用 MongoDB 最合适 |
| **评论系统** | ⭐⭐⭐⭐ | 树形结构，内嵌方便，结构灵活 | 无 | 开发更快，查询性能更好 |
| **消息通知** | ⭐⭐⭐⭐ | 结构多变，不同类型不同字段 | 无 | 灵活扩展，适应需求变化 |
| **访客UV/PV统计** | ⭐⭐⭐⭐ | 写入频繁，允许最终一致 | 无 | 发挥 MongoDB 原子更新优势 |
| **文章草稿** | ⭐⭐⭐ | 结构可变，不需要强一致 | 无 | 适应编辑器需求变化 |
| **操作审计日志** | ⭐⭐⭐ | 写入多，读取少，结构灵活 | 无 | 不占用核心数据库空间 |
| **点赞/收藏关系** | ⭐⭐ | 写入频繁，独立查询 | MySQL | 提升写入性能，减轻 MySQL 压力 |

## 总结：MongoDB 适合 vs 不适合

### ✅ 适合 MongoDB 的特征

1. **写入量大，读取相对少** - 日志、审计、行为记录
2. **文档型/树形结构** - 文章内容、评论
3. **结构多变，需要频繁扩展** - 通知、草稿、用户画像
4. **对一致性要求不是极端强** - 统计数据、行为数据
5. **水平扩展预期大** - 历史记录、日志数据

### ❌ 不适合 MongoDB（建议留在 MySQL）

1. **需要强事务一致性** - 用户账户、认证信息、余额相关
2. **需要频繁多表关联查询** - 分类列表、文章元数据关联分类
3. **需要严格唯一性约束且不能容忍任何错** - 用户账号、唯一用户名
4. **数据结构稳定，关系清晰** - 系统配置、基础数据字典

## 渐进式引入建议

如果你决定引入 MongoDB，推荐路径：

1. **第一步**：日志迁移到 MongoDB（方案一），验证基础设施和团队使用经验
2. **第二步**：文章内容块迁移到 MongoDB（方案二），感受性能提升
3. **第三步**：后续开发**新功能**直接用 MongoDB（浏览历史、评论、通知、统计等）
4. **第四步**：观察一段时间，效果好再考虑逐步迁移其他适合的数据

这样风险最小，收益逐步体现，团队可以逐步适应。
