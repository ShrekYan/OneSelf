# MongoDB 接入方案对比

## Context

当前项目是全栈博客项目，采用 Monorepo 微服务架构：
- **核心业务数据**（用户、文章、分类、点赞等）存储在 **MySQL**，使用 Prisma ORM 访问
- **日志数据**存储在本地文件，由 log-service 独立处理
- 业务形态：博客文章发布和阅读，以读为主

用户想了解项目是否需要接入 MongoDB，以及可以在哪些地方改造，需要对比不同方案后选择。

---

## 三个接入方案对比

### 方案一：MongoDB 仅用于日志存储（最小侵入）

**思路**：保持核心业务全部在 MySQL，只把日志从文件迁移到 MongoDB。侵入性最小，风险最低。

**迁移范围**：

| 数据 | 是否迁移 | 原因 |
|------|---------|------|
| 用户认证 | ❌ 不迁移 | 强一致性要求，MySQL 更适合 |
| 文章分类 | ❌ 不迁移 | 结构化关联数据，MySQL 足够 |
| 文章内容 | ❌ 不迁移 | 当前分块存储稳定，不需要改动 |
| **访问日志** | ✅ 迁移 | 半结构化、写入量大、不需要事务，MongoDB 更适合 |

**架构变化**：
```
┌─────────────┐     ┌──────────────┐
│  backend    │────▶│  MySQL       │ ← 不变
└─────────────┘     └──────────────┘
        ↑
┌─────────────┐
│ auth-service│────▶ MySQL  ← 不变
└─────────────┘
        ↑
┌─────────────┐     ┌──────────────┐
│ log-service │────▶│  MongoDB     │ ← 日志迁移
└─────────────┘     └──────────────┘
```

**工作量**：**2-3 人天**，仅需改造 log-service，其他服务不动。

**优点**：
- ✅ 侵入性极小，不影响核心业务
- ✅ 风险极低，出错不影响博客主流程
- ✅ 改造成本低，开发速度快
- ✅ 可以验证 MongoDB 运维流程，团队试水

**缺点**：
- ❌ 只用到 MongoDB 最基础能力，核心业务没有受益
- ❌ 还是需要维护多数据源

**适用**：团队首次引入 MongoDB，需要试水；当前日志文件存储遇到问题。

---

### 方案二：混合存储架构（部分迁移，推荐）

**思路**：核心关系数据保持 MySQL，非结构化文档数据迁移 MongoDB。充分发挥两种数据库各自优势。

**迁移范围**：

| 数据 | 是否迁移 | 存储 | 原因 |
|------|---------|------|------|
| 用户认证 | ❌ | MySQL | 需要事务一致性 |
| 分类 | ❌ | MySQL | 关联查询需要 |
| 文章元数据 | ❌ | MySQL | 标题/摘要/统计需要索引 |
| 点赞关系 | ❌ | MySQL | 需要唯一约束防重复 |
| **文章内容块** | ✅ | MongoDB | 文档型结构灵活，内嵌更自然 |
| **访问日志** | ✅ | MongoDB | 和方案一相同 |

**为什么文章内容适合 MongoDB**：
- 当前 MySQL 需要拆分多行存储内容块，查询需要 JOIN
- MongoDB 可以一篇文章一个文档，内嵌所有内容块，一次查询获取完整文章
- 未来扩展块类型（视频、卡片等）不需要 ALTER TABLE

**架构变化**：
```
┌─────────────────────────────────────────┐
│                 backend                 │
│                                         │
│  ┌─────────────┘    ┌──────────────┐    │
│  │ 用户/分类/点赞  │─▶  MySQL      │    │
│  │  (关系数据)    │  │ (核心结构化)│    │
│  └─────────────────┘  └──────────────┘    │
│                                         │
│  ┌─────────────────┘    ┌──────────────┐    │
│  │ 文章内容/日志    │─▶  MongoDB     │    │
│  │  (文档数据)     │  │ (非结构化)   │    │
│  └─────────────────┘  └──────────────┘    │
└─────────────────────────────────────────┘
        ↑
┌─────────────┐
│ auth-service│────▶ MySQL  ← 不变
└─────────────┘
        ↑
┌─────────────┐
│ log-service │────▶ MongoDB (复用连接)
└─────────────┘
```

**工作量**：**7-10 人天**，需要改造 article 模块，开发数据迁移脚本。

**优点**：
- ✅ 扬长避短，各得其所：MySQL 处理关系，MongoDB 处理文档
- ✅ 查询性能提升：一次查询获取整篇文章，减少 JOIN
- ✅ 结构扩展灵活：新增内容块类型不需要改表
- ✅ 核心数据仍在 MySQL，保持事务保证
- ✅ 风险可控，渐进式迁移

**缺点**：
- ❌ 需要同时维护双数据源，代码复杂度增加
- ❌ 需要处理跨库数据一致性
- ❌ 运维成本增加，需要监控备份两个数据库

**适用**：希望利用 MongoDB 优势，但不完全放弃 MySQL；这是推荐方案，风险收益比最高。

---

### 方案三：全量迁移（完全替换）

**思路**：移除 MySQL，所有核心业务数据全部迁移到 MongoDB。彻底切换，统一存储。

**迁移范围**：**所有数据全部迁移**到 MongoDB。

- 用户信息 → MongoDB
- 文章分类 → MongoDB
- 文章元数据 + 内容 → MongoDB（一篇文章一个文档）
- 点赞关系 → MongoDB
- JWT 刷新令牌 → MongoDB
- 热搜关键词 → MongoDB
- 日志 → MongoDB

**为什么全量迁移**：
- 统一数据库，降低运维复杂度
- MongoDB 文档模型更贴合博客"文章为中心"的业务形态
- Prisma 6+ 已经支持 MongoDB，开发体验可以保持
- 未来业务扩展更灵活

**架构变化**：
```
┌─────────────┐
│  backend    │────┐
└─────────────┘    │
┌─────────────┐    │
│ auth-service│────┼────▶ MongoDB（所有数据）
└─────────────┘    │
┌─────────────┐    │
│ log-service │────┘
└─────────────┘
```

**工作量**：**15-20 人天**，大范围改造，需要全量回归测试。

**优点**：
- ✅ 统一存储，架构简洁，运维简单
- ✅ 充分发挥 MongoDB 文档模型优势
- ✅ schema 灵活，适应未来业务变化
- ✅ 查询性能最优，一篇文章一次读取

**缺点**：
- ❌ 迁移工作量巨大，风险高
- ❌ 事务支持不如 MySQL，点赞关系一致性处理更复杂
- ❌ 需要重新设计数据模型，容易引入 bug
- ❌ 全量回归测试成本很高

**适用**：团队熟悉 MongoDB，希望彻底切换；业务快速迭代需要 schema 极高灵活性。

## 方案对比汇总

| 对比维度 | 方案一：仅日志 | 方案二：混合存储 | 方案三：全量替换 |
|---------|---------------|-----------------|-----------------|
| **侵入程度** | 最小 | 中等 | 完全改造 |
| **改造工作量** | 2-3 人天 | 7-10 人天 | 15-20 人天 |
| **风险等级** | 极低 | 中等 | 高 |
| **架构复杂度** | 略增 | 中等（双数据源） | 统一简化 |
| **发挥 MongoDB 优势** | 很少 | 部分（扬长避短） | 充分 |
| **核心事务一致性** | 不受影响 | 核心保证 | 应用层处理 |
| **运维成本** | 新增一个实例 | 两个数据库都要维护 | 只维护一个 |
| **适合阶段** | 试点探索 | 生产可用推荐 | 彻底重构 |

---

## 推荐方案：方案二（混合存储）

**推荐理由**：

1. **最贴合业务实际**：博客本质就是"关系数据 + 文档内容"的混合形态，关系数据用 MySQL 保证一致性，文档内容用 MongoDB 获取灵活性，这是最合理的划分。

2. **风险收益比最高**：相比方案一收获更大，相比方案三风险可控。既能验证 MongoDB 使用，又不会一下子承担过大风险。

3. **渐进式迁移路径**：即使未来打算全量迁移，也可以先走方案一再到方案二，积累经验后再逐步推进，步子更稳。

**渐进实施路线图**：
1. 先实施**方案一**：日志迁移 MongoDB，验证基础设施
2. 再实施**文章内容迁移**，完成方案二
3. 观察稳定性和性能收益
4. 若体验良好，再考虑逐步迁移其他模块

## 关键文件（按方案）

### 方案一（仅日志）
- `services/log-service/src/logger/file-logger.ts` - 需要改造为 MongoDB 存储
- `services/log-service/src/app.module.ts` - 新增 MongoDB 模块配置

### 方案二（混合存储）
- `services/backend/prisma/schema.prisma` - 新增 MongoDB 数据源，移除 ArticleContentBlocks
- `services/backend/src/article/article.service.ts` - 改造文章查询/写入逻辑
- `services/backend/src/app.module.ts` - 新增 MongoDB 连接配置
- 数据迁移脚本：从 MySQL 迁移内容到 MongoDB

### 方案三（全量替换）
- `services/*/prisma/schema.prisma` - 所有服务修改数据源为 MongoDB
- 所有服务数据模型需要调整适配 Document 模型
- 全量数据迁移脚本

## 验证方法

实施后验证步骤：
1. 运行 `npm run lint` 检查代码风格
2. 服务启动验证：各个服务能正常连接数据库
3. 功能验证：对应模块API接口测试
4. 数据一致性验证：迁移后数据完整性检查
5. 性能验证：对比改造前后查询耗时
