# Claude Code Agent Skill 加载机制深度解析

## 📌 问题背景

在自定义 Agent 配置时，我们最初尝试通过**文字说明 + 代码块**的方式要求 Agent 在开始任务前先加载 Skill：

```markdown
# ⚠️ 第一步：必须执行（不执行就不能开始）

在开始任何涉及 NestJS 代码开发的任务之前，**你必须立即执行以下命令**：
```
/skill nestjs-backend-developer
```
```

**实际效果**：这种方式的生效准确率大约只有 **30%**。

---

## 🔍 根本原因分析

### 1. Agent Prompt 的执行逻辑

Claude Code Agent 被激活时：
- ✅ 会读取完整的 Agent 配置文件作为系统提示词
- ❌ **不会自动扫描并执行 "第一步必须做什么" 这类自然语言指令**
- ❌ 代码块（```）包裹的内容会被当作普通文本/示例处理
- ❌ 依赖大模型"阅读理解"后主动执行是不可靠的

### 2. Skill 工具的调用机制

`/skill xxx` 命令需要：
1. 明确调用 `Skill` 工具
2. 传入 skill 名称参数
3. 等待工具返回结果后，规范内容才真正进入上下文

**自然语言指令无法保证 100% 触发工具调用**，因为：
- Agent 可能"理解"了要求，但认为"我已经知道了"
- Agent 可能跳过这一步直接响应用户需求
- 多轮对话中上下文稀释导致遗忘

### 3. 系统提示词 vs 动态 Skill 加载

| 方式 | 时机 | 可靠性 | 说明 |
|------|------|--------|------|
| **Agent 文件直接写规范** | 启动时加载 | ✅ 100% | 永久生效，无遗漏 |
| **`#include:` 指令** | 启动时解析 | ✅ 100% | 同文件直接写，支持模块化 |
| **运行时 `/skill` 调用** | 对话开始后 | ⚠️ 30-70% | 依赖 Agent 主动执行，不稳定 |

---

## ✅ 最佳实践方案

### 方案：使用 `#include:` 指令内联 Skill

```markdown
---
name: backend-architect
description: ...
tools: Read, Write, Edit, Skill
model: inherit
---

<!-- 🔐 项目规范自动加载 - 100% 生效，无需手动调用 -->
#include: ../skills/nestjs-backend-developer/nestjs-backend-developer.md

---

# 角色定位
...
```

**工作原理**：
1. Claude Code 启动 Agent 时会**预处理**配置文件
2. 识别 `#include:` 指令，自动读取对应文件内容
3. 将 Skill 内容**直接内联**到系统提示词中
4. Agent 从第一秒就拥有完整的项目规范知识

**优势**：
- ✅ 零遗漏，100% 加载到上下文
- ✅ 无需 Agent 主动执行任何操作
- ✅ 模块化管理，Skill 更新自动同步
- ✅ 减少对话轮次，提高响应速度

## 🚫 常见反模式

### ❌ 反模式 1：依赖自然语言提醒

```markdown
<!-- 不可靠！！！ -->
> 重要：这是强制性的。只有加载了完整的项目规范后，你才能理解本项目的约定。
```

**问题**：依赖 Agent "自觉"执行，稳定性完全不可控。

### ❌ 反模式 2：代码块包裹示例命令

```markdown
<!-- 不可靠！！！ -->
```
/skill nestjs-backend-developer
```
```

**问题**：代码块通常被理解为"示例"，而不是"必须执行的指令"。

### ❌ 反模式 3：Skill 只在 tools 列表声明

```yaml
tools: Read, Write, Edit, Skill  # ❌ 只是声明可用，不会自动加载
```

**问题**：只是告诉 Agent 可以用 Skill 工具，不会自动加载任何特定 Skill。

## 🎯 适用场景

### 必须用 `#include:` 内联的场景
1. ✅ 项目级强制编码规范
2. ✅ 技术栈特定开发约定
3. ✅ 目录结构、命名规范
4. ✅ 代码审查检查清单

### 可以用动态 `/skill` 的场景
1. ⚠️ 可选的辅助工具（如代码审查、性能分析）
2. ⚠️ 任务中后期才需要的专项能力
3. ⚠️ 用户明确触发的一次性操作

### 关键修复点详解

1. **`#` 前缀统一**
   - 问题：`include:` 无 `#` 前缀被当作普通文本
   - 修复：全部改为 `#include:`，触发预处理机制

2. **相对路径修正**
   - 问题：前端 Skill 中错误地写了 `h5-frontend-developer/xxx.md`
   - 原因：`#include:` 的基准目录是**当前文件所在目录**
   - 修复：改为 `architecture-directory.md`（同级文件直接写）

3. **递归解析链完整化**
   ```
   修复前：Agent → Skill 主文件(停止) → 子文件❌
   修复后：Agent → Skill 主文件 → 子文件 → rules/ 子目录 ✅
   ```

## 💡 进阶技巧

### 1. 部分 Skill 按需动态加载

对于大型 Agent，可以将规范分层：

```markdown
---
name: full-stack-architect
---

<!-- 核心规范：100% 内联 -->
#include: ../skills/core-project-rules.md

<!-- 可选规范：按需触发 -->
<!-- 如需后端开发，请执行 /skill nestjs-backend-developer -->
<!-- 如需前端开发，请执行 /skill h5-frontend-developer -->
```

### 2. 验证 Skill 是否正确加载

在 Agent 响应流程中添加检查：

```markdown
# 响应流程

0. **确认规范已加载** - 在开始前默念一遍项目的目录结构规范
1. 理解需求...
```

### 3. Skill 嵌套引用

Skill 内部也可以继续用 `#include:` 引用子模块：

```markdown
// nestjs-backend-developer.md
#include: 01-architecture-module.md
#include: 02-file-naming.md
#include: 03-controller-service.md
```

## 📋 检查清单

创建新 Agent 配置时，请确认：

- [ ] 核心项目规范已通过 `#include:` 内联到 Agent 文件头部
- [ ] 没有用"请先执行 `/skill xxx`"这类自然语言说明
- [ ] Skill 文件路径正确，相对路径基于 Agent 位置
- [ ] Skill 内容不超过上下文窗口限制
- [ ] 动态加载的 Skill 只用于可选/辅助功能

---

## 🎓 总结

**核心原则**：**凡是 Agent 必须遵守的规范，一定要直接在系统提示词中存在，不能依赖 Agent 动态加载。**

Agent 系统提示词是 Claude Code 中最稳定、最可靠的知识来源。任何需要 100% 遵守的约定，都应该在启动时就固化到上下文中，而不是寄希望于运行时动态获取。
