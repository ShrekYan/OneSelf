### 1.2 这个提示词好在哪里？

> 💡 **教学重点**：好的提示词 = 「现象 + 边界 + 标准 + 安全」。这正是架构师解决问题的标准思维流程。

### 1.4 如果用户没说这些，会发生什么？（反事实推演）

| 如果您没说这句话 | AI 大概率会怎么做 | 后果 | 需求的价值 |
|------------------|-------------------|------|------------|
| ❌ 没说"找到根本原因" | 给首页加个 `useEffect`，每次挂载清空所有状态 | 页面闪烁、性能下降、用户体验变差，留下隐形技术债 | ⭐⭐⭐⭐⭐ 避免了暴力 hack 式修复，保护了架构的干净 |
| ❌ 没说"不要影响其他功能" | 直接关闭 KeepAlive 缓存，一了百了 | 首页每次进入都要重新加载，滚动位置丢失，用户体验大幅下降 | ⭐⭐⭐⭐ 保留了缓存功能的价值，在用户体验和修复间找到平衡 |
| ❌ 没说"不需要传递 userId" | 前端继续传 userId，后端继续用 body 里的 userId | 越权漏洞，用户 A 可以修改用户 B 的点赞状态，严重安全风险 | ⭐⭐⭐⭐⭐ 堵住了 OWASP Top 10 级别的安全漏洞 |
| ❌ 没说"仅更改css样式就行" | 去改 JS 逻辑，加点击后强制重绘，用 setState 刷新 | 引入新的状态管理复杂度，可能导致额外的 rerender 性能问题 | ⭐⭐⭐ CSS 层问题在 CSS 层解决，分层原则 |
| ❌ 没有迭代优化提示词 | AI 一直在错误的方向上修来修去，反复迭代 | 浪费时间，多次沟通成本，甚至可能引入新的 bug | ⭐⭐⭐⭐ 好的沟通是高效协作的基础 |

> 🎓 **教学总结**：每一句约束都是在引导 AI 走向正确的架构决策，每一次提示词迭代都是在降低沟通成本。

## 3. 问题一：跨用户点赞状态残留

### 3.1 根因分析流程图

```
用户A点赞文章
    ↓
isLiked = true 写入 MobX store
    ↓
用户A退出登录
    ↓ （问题关键点）
KeepAlive 缓存页面，store 实例不销毁 ←──────┐
    ↓                                         │
用户B登录并进入首页                            │
    ↓                                         │
fetchInitialData() 开始执行                    │
    ↓                                         │
┌─────────────────────────────────────────────┤
│ ❌ 旧的 likedArticleIds 仍然存在！           │
└─────────────────────────────────────────────┤
    ↓
fetchArticles 获取用户B的文章列表
    ↓
文章列表里没有 isLiked 字段（后端不返回）
    ↓
applyLikeStatusToArticles()
    ↓
💥 爆炸：用户B看到的是用户A的点赞状态！
```

### 3.2 修复方案对比

| 方案 | 具体做法 | 优点 | 缺点 | 用户的选择 |
|------|---------|------|------|-----------|
| 方案 A：暴力清缓存 | 退出登录时销毁所有 KeepAlive 页面 | 简单粗暴，一劳永逸 | 牺牲用户体验，所有页面滚动位置丢失 | ❌ 拒绝 |
| 方案 B：监听用户变化 | store 里用 useEffect 监听 userInfo 变化 | 响应式，自动清除 | 增加复杂度，有竞态风险 | ❌ 拒绝 |
| 方案 C：数据获取前重置 | 在 fetchInitialData 最开头清空状态 | ✅ 不影响缓存功能<br>✅ 逻辑集中，数据流起点重置<br>✅ 无竞态风险 | 无明显缺点 | ✅ **用户选择** |

> 💡 **架构师思维**：状态重置应该放在数据流的起点，而不是放在"退出登录"这种业务边界点。因为数据流的起点是唯一的，而业务边界可能有很多个（token 过期、被挤下线、切换账号...），容易漏。

### 3.3 代码实现

**修改前（有问题）**：
```typescript
// apps/web/src/pages/Discover/routes/home/useStore.ts
async fetchInitialData(): Promise<void> {
  // ❌ 没有清空旧状态，直接开始获取数据
  const userId = rootStore.app.userInfo?.id;

  // 直接获取数据...
  const results = await Promise.allSettled([...]);

  // ...

  // ❌ 只有当 likedArticleIds 有内容时才应用状态
  if (this.likedArticleIds.size > 0) {
    this.applyLikeStatusToArticles();
  }
}
```

**修改后（正确）**：
```typescript
// apps/web/src/pages/Discover/routes/home/useStore.ts
async fetchInitialData(): Promise<void> {
  // ✅ 修复：在获取数据前先清空旧的点赞状态，避免跨用户残留
  runInAction(() => {
    this.likedArticleIds = new Set();
  });

  const userId = rootStore.app.userInfo?.id;

  // ...

  // ✅ 修复：无条件执行，空集合会正确清除所有文章的 isLiked 标记
  this.applyLikeStatusToArticles();
}
```

> 💡 **架构师思维**：条件判断 `if (size > 0)` 看起来是"优化"，实际上是隐藏的 bug。当新用户没有任何点赞时，旧状态就永远不会被清除。很多时候，"少做就是多做"。

## 4. 问题二：后端 user-likes 接口安全加固

### 4.1 安全漏洞分析

```
前端请求链路信任分析
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  ┌─────────────┐         ┌─────────────┐         ┌─────────────┐  │
│  │   前端页面   │ ──────> │   后端接口   │ ──────> │    数据库    │  │
│  │  传 userId  │         │  用 body 的  │         │             │  │
│  │             │         │   userId    │         │             │  │
│  └─────────────┘         └─────────────┘         └─────────────┘  │
│         │                      │                      │              │
│         ▼                      ▼                      ▼              │
│   用户可以篡改！❌        不可信源！❌          数据污染！❌         │
│                                                                     │
│  攻击方式：                                                         │
│  1. 打开浏览器控制台                                                │
│  2. 修改请求体中 userId 为其他用户 ID                                │
│  3. 查看任意用户的点赞列表                                          │
│                                                                     │
│  漏洞等级：严重 ⚠️⚠️⚠️                                               │
│  修复成本：3 行代码                                                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 修复方案对比

| 方案 | 具体做法 | 优点 | 缺点 | 用户的选择 |
|------|---------|------|------|-----------|
| 方案 A：后端校验 | 校验 body.userId 与 JWT.userId 一致 | 不修改前端 | 逻辑冗余，既然有 JWT 为什么还要传？ | ❌ 不优雅 |
| 方案 B：完全信任 JWT | 从 @CurrentUserId 获取，不传 body 参数 | ✅ 最安全，最简洁<br>✅ JWT 是唯一可信源<br>✅ 前端代码更简洁 | 前后端需要一起改 | ✅ **用户选择** |

> 🔒 **安全思维**：零信任原则 - 永远不要信任前端传来的用户标识。JWT 是唯一可信的用户身份来源。

### 4.3 代码实现

**Controller 修改前（有问题）**：
```typescript
// services/backend/src/article/article.controller.ts
@UseGuards(RemoteJwtAuthGuard)
@Post('user-likes')
getUserLikeListByUserId(
  @Body() body: UserLikeListByUserIdRequestDto,  // ❌ 信任前端传来的 userId
): Promise<UserLikeListByUserIdResponseDto> {
  return this.articleService.getUserLikeListByUserId(body);
}
```

**Controller 修改后（正确）**：
```typescript
// services/backend/src/article/article.controller.ts
@UseGuards(RemoteJwtAuthGuard)
@Post('user-likes')
getUserLikeListByUserId(
  @CurrentUserId() userId: string,  // ✅ 从认证系统获取，唯一可信源
): Promise<UserLikeListByUserIdResponseDto> {
  return this.articleService.getUserLikeListByUserId({ userId });
}
```

**前端 API 修改后**：
```typescript
// apps/web/src/api/article/index.ts
getUserLikeList: async (): Promise<UserLikeListByUserIdResponse> => {
  return await api.post('/api/v1/article/user-likes');  // ✅ 不传参数
},
```

> 💡 **架构师思维**：安全的接口设计应该是"最小依赖原则"——前端不需要传的参数，就不要传。参数越少，攻击面越小。

## 5. 问题三：CSS 取消点赞颜色状态管理

### 5.1 根因分析流程图

```
┌─────────────────────────────────────────────────────────────────────┐
│                    CSS 选择器优先级问题分析                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  默认样式：.statBtn svg { fill: currentcolor; }                      │
│       作用于 svg 元素                                                 │
│           │                                                          │
│           ▼                                                          │
│  激活样式：.statBtn.active svg path { fill: #ff4d4f; }              │
│       作用于 svg path 元素（更具体）                                  │
│           │                                                          │
│           ▼                                                          │
│  💥 问题出现：                                                        │
│  1. 用户点赞 → .active 类加上 → path 的 fill 变成红色               │
│  2. 用户取消点赞 → .active 类移除                                    │
│  3. 但是！没有任何样式来重置 path 的 fill！！！                       │
│  4. 因为默认样式作用于 svg，优先级不够，覆盖不了 path 上的属性        │
│                                                                     │
│  结果：取消点赞后，心形图标永远是红色！😭                             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.2 修复方案对比

| 方案 | 具体做法 | 优点 | 缺点 | 用户的选择 |
|------|---------|------|------|-----------|
| 方案 A：JS 强制重绘 | 点击后加 key 让 React 重新渲染组件 | 能解决问题 | 太暴力，CSS 问题用 JS 解决，不优雅 | ❌ 拒绝 |
| 方案 B：CSS !important | 默认样式加 !important 提高优先级 | 简单粗暴 | 优先级战争的开始，后续维护噩梦 | ❌ 拒绝 |
| 方案 C：选择器层级对齐 | 默认样式也作用于 svg path，与激活状态同层级 | ✅ 最优雅，CSS 问题 CSS 层解决<br>✅ 无副作用<br>✅ 符合 CSS 设计原则 | 无缺点 | ✅ **用户选择** |
| 方案 D：移除 hover 颜色 | 未点赞状态 hover 不显示紫色 | ✅ 符合用户期望（用户明确要求）<br>✅ 代码最简洁 | 减少了一点交互反馈 | ✅ **用户也选择** |

> 💡 **架构师思维**：CSS 也有架构原则。相同属性的默认样式和激活样式，必须作用于相同的 DOM 层级。这是 CSS 可维护性的基础原则。

### 5.3 代码实现

**第一处修复：选择器层级对齐**

```scss
// 修改前（有问题）
.statBtn {
  svg {
    width: 24px;
    height: 24px;
    fill: currentcolor;  // ❌ 作用于 svg，层级不一致
  }
}

// 修改后（正确）
.statBtn {
  svg {
    width: 24px;
    height: 24px;
    // ✅ 移除 fill，移到下面的 path 选择器
  }

  svg path {
    fill: currentcolor;  // ✅ 与激活状态相同层级
  }
}
```

**第二处修复：移除未点赞的 hover 紫色**

```scss
// 修改前（有问题）
.statBtn {
  &:hover {
    color: #8848f9;  // ❌ 未点赞 hover 也变紫色
  }
}

// 修改后（正确）
.statBtn {
  // ✅ 移除了未点赞状态的 hover 紫色

  &.active:hover {
    color: #ff4d4f;  // ✅ 只有已点赞的 hover 保持红色
  }
}
```

> 💡 **架构师思维**：交互设计原则——当用户的操作结果是"取消"时，视觉上应该是"回归默认"，而不是"变成另一种高亮状态"。取消点赞后鼠标还在按钮上，此时应该显示灰色，告诉用户"你已经成功取消了"。

## 8. 教学总结

### 8.1 学到的三个核心原则

| 原则 | 具体内容 | 本次案例中的体现 |
|------|---------|----------------|
| 🔍 **根因原则** | 问题在哪层就在哪层解决，不要跨层 | CSS 问题 CSS 解决，状态问题在数据流起点解决 |
| 🔒 **零信任原则** | JWT 是唯一可信的用户身份源，永远不要信任前端传来的 userId | user-likes 接口从 body 改到 @CurrentUserId |
| 🎯 **分层原则** | 相同属性的样式/状态，必须作用于相同的层级 | svg path 选择器对齐，相同属性不能跨层级 |

### 8.2 常见踩坑点总结

| 踩坑点 | 现象 | 避免方法 |
|--------|------|---------|
| ❌ KeepAlive 状态残留陷阱 | 切换用户后旧页面的状态还在 | 所有依赖用户身份的状态，都在数据获取前清空 |
| ❌ 信任前端传参陷阱 | userId 从 body 传 | 所有用户相关操作，只用 @CurrentUserId，别信前端 |
| ❌ CSS 层级陷阱 | 相同属性作用于不同层级 | 默认样式和激活样式必须作用于相同的 DOM 层级 |
| ❌ "优化"反而引入 bug | `if (size > 0)` 看起来是优化 | 少做就是多做，不要加不必要的条件判断 |
