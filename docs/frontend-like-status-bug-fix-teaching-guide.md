## 3. 问题一：跨用户点赞状态残留

### 3.1 根因分析流程图

```
用户A点赞文章
    ↓
isLiked = true 写入 MobX
    ↓
用户A退出登录
    ↓ （问题点）
KeepAlive 缓存页面，store 不销毁 ←─────┐
    ↓                                    │
用户B登录首页                              │
    ↓                                    │
fetchInitialData()                       │
    ↓                                    │
isLiked 字段残留，新用户无点赞            │
    ↓                                    │
显示用户A的点赞状态！ ◄──────────────────┘
```

### 3.2 修复方案对比

| 方案 | 做法 | 优点 | 缺点 | 您的选择 |
|------|------|------|------|---------|
| 方案 A：暴力清除 | 退出时销毁页面缓存 | 简单粗暴 | 牺牲用户体验，滚动位置丢失 | ❌ |
| 方案 B：监听用户ID | store 监听 userInfo 变化清除 | 响应式 | 增加复杂度，有竞态风险 | ❌ |
| 方案 C：数据获取前重置 | fetchInitialData 开头清空状态 | ✅ 不影响缓存，逻辑集中，与数据刷新天然同步 | 无明显缺点 | ✅ **您的选择** |

### 3.3 代码实现

**修改前（有问题）**：
```typescript
// apps/web/src/pages/Discover/routes/home/useStore.ts
async fetchInitialData(): Promise<void> {
  const userId = rootStore.app.userInfo?.id;
  // 直接获取数据，旧状态残留 ❌
  const results = await Promise.allSettled([...]);
}
```

**修改后（正确）**：
```typescript
// apps/web/src/pages/Discover/routes/home/useStore.ts
async fetchInitialData(): Promise<void> {
  // ✅ 修复：在获取数据前先清空旧的点赞状态，避免跨用户残留
  runInAction(() => {
    this.likedArticleIds = new Set();
  });

  const userId = rootStore.app.userInfo?.id;
  // ...
}
```

> 💡 **架构师思维**：状态重置应该放在数据获取的最开头，这是数据流的起点，保证所有后续操作基于干净的状态。

## 4. 问题二：后端接口安全加固

### 4.1 安全漏洞分析

```
前端请求链路
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   前端页面   │ ──────> │   后端接口   │ ──────> │    数据库    │
│  传 userId  │         │  用 body 的  │         │             │
│             │         │   userId    │         │             │
└─────────────┘         └─────────────┘         └─────────────┘
       │                      │                      │
       ▼                      ▼                      ▼
  用户可以篡改！❌        不可信源！❌        数据污染！❌
```

### 4.2 修复方案

| 层级 | 修改前 | 修改后 |
|------|--------|--------|
| **Controller** | `@Body() body: UserLikeListByUserIdRequestDto | `@CurrentUserId() userId: string` |
| **Service** | 依赖 DTO 验证 | 直接使用认证系统的 userId |
| **前端 API** | `post(url, { userId }) | `post(url)` 不传参 |

> 🔒 **安全思维**：JWT 是唯一可信的用户身份来源，永远不要信任前端传来的用户标识。

## 5. 问题三：CSS 颜色状态管理

### 5.1 CSS 选择器优先级问题

```
选择器优先级对比
┌─────────────────────────────────────────────────────────────┐
│                                                           │
│  默认样式：.statBtn svg { fill: currentcolor; }             │
│       作用于 svg 元素                                         │
│                                                           │
│  激活样式：.statBtn.active svg path { fill: #ff4d4f; }    │
│       作用于 svg path 元素                                   │
│                                                           │
│  问题：.active 移除后，path 的 fill 没有被重置！            │
│                                                           │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 修复代码

**修改前（有问题）**：
```scss
.statBtn {
  svg {
    fill: currentcolor;  // ❌ 作用层级不一致
  }
  &.active svg path {
    fill: #ff4d4f;
  }
}
```

**修改后（正确）**：
```scss
.statBtn {
  svg {
    width: 24px;
    height: 24px;
  }
  svg path {
    fill: currentcolor;  // ✅ 与激活状态相同层级
  }
}
```

> 🎨 **CSS 架构思维**：相同属性的默认样式必须作用于相同的 DOM 层级，这是 CSS 状态管理的基本原则。

## 8. 教学总结

### 8.1 学到的三个核心原则

| 原则 | 具体内容 | 本次案例中的体现 |
|------|---------|----------------|
| **🔍 根因原则 | 现象 → 链路 → 根因 | 不暴力清除缓存，而是在数据刷新源头重置状态 |
| **🔒 安全原则 | JWT 是唯一可信源 | 所有用户相关操作都从认证系统获取，不信任前端 |
| **🎯 分层原则 | 问题在哪层就在哪层解决 | CSS 问题 CSS 层解决，不越界到 JS |

### 8.2 常见踩坑点

1. ❌ **状态残留陷阱**：缓存页面的 store 不会销毁，退出登录时要考虑所有缓存页面的状态
2. ❌ **信任前端传参陷阱**：不要信任前端传来的 userId，JWT 才是唯一可信的
3. ❌ **CSS 层级陷阱**：相同属性的默认样式必须作用于相同的 DOM 层级
