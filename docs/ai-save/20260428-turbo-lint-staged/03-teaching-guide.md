## 💡 核心知识点沉淀

### 知识点 1：lint-staged 的工作原理

**核心理解**：
```
lint-staged 在哪里执行，就用哪里的规则！

在 apps/web 目录执行：
  → eslint 从 apps/web 向上找配置文件
  → 匹配 apps/web/eslint.config.js
  → glob 是相对路径，只匹配 apps/web/src/...

在 services/backend 目录执行：
  → 匹配 services/backend/eslint.config.mjs
  → glob 只匹配 services/backend/src/...
```

**关键结论**：
- 虽然 git diff 拿到的是全仓库 staged 文件列表
- 但每个子项目的 lint-staged 只会挑自己目录下的文件
- 用自己的 eslint 规则检查
- 同一个文件永远只会被一个子项目匹配到

---

### 知识点 2：什么时候该用第三方插件？

**判断标准（满足 2 条以上就用）**：
1. ✅ 这个问题已经存在 5 年以上
2. ✅ GitHub Star 超过 10k
3. ✅ 周下载量超过 100 万
4. ✅ 你自己写需要超过 50 行代码
5. ✅ 涉及跨平台兼容、边缘 case 处理

**本次案例**：
- lint-staged: 12k+ Star, 1000 万+ 周下载, 存在 8 年
- 满足全部 5 条，直接用，不要自己写

**反例**：
- 一个 3 行的工具函数 → 自己写就行，不用找 npm 包
- 业务逻辑非常特殊的场景 → 自己写

---

### 知识点 3：Monorepo 的任务编排原则

| 原则 | 说明 | 本次应用 |
|------|------|---------|
| **子项目自治** | 子项目的配置放在子项目里，不要都堆在根目录 | ✅ 每个子项目自己的 lint-staged 配置放在自己的 package.json |
| **根目录只做编排** | 根目录只负责"调用"，不负责"具体实现" | ✅ 根目录只有 `turbo lint:staged`，具体实现各子项目自己管 |
| **并行优先** | 能并行的任务不要串行 | ✅ Turbo 自动并行 4 个子项目 |
| **增量执行** | 没有变更的子项目快速跳过 | ✅ 无 staged 文件的子项目输出"No staged files"直接退出 |

---

### 知识点 4：如何优雅地推翻 AI 的方案？

**正确姿势（本次演示得非常好）**：

**不要说**：❌ "你这个方案不行，换一个"
**要说**：✅ "我要优化一点：xxx，难道不可以用 yyy？搞的这么复杂，后期怎么维护？"

**为什么效果好**：
1. 直接点出问题的本质（维护成本）
2. 给出了更好的方向（第三方插件）
3. 用提问的方式，不是否定，AI 更容易接受
4. 体现了你自己的思考，不是被动接受

**结果**：AI 立刻意识到问题，推翻了自己之前的方案，给出了更优的方案。

## ⚠️ 踩坑教训记录

### 踩坑 1：不要自己实现成熟插件的功能

**错误现象**：
- 第一轮方案写了 200 多字符的 Node.js 内联脚本
- 包含 git 命令调用、正则过滤、工具执行、错误处理
- 非常难读，新人根本看不懂

**根本原因**：
- 陷入"我要实现这个功能"的思维
- 没有先跳出来想"是不是已经有人把这个问题解决了？"

**正确做法**：
- 先想 3 秒：这是通用问题吗？有没有成熟方案？
- lint-staged 就是专门干这个的，直接用就行

---

### 踩坑 2：不要忽略方案的维护成本

**错误现象**：
- AI 给出的方案"能工作"，但维护成本很高
- 复杂的转义、正则、内联脚本
- 出了问题没人能 debug

**根本原因**：
- 只考虑"功能能不能实现"，没考虑"以后好不好维护"
- 技术选型只看功能，不看团队能力

**正确做法**：
- 任何方案都要加"维护成本"这个评估维度
- 能用成熟生态解决的，就不要自己发明
- 团队里每个人都能看懂的方案，才是好方案
