```markdown
# Agent 自动触发机制优化任务

## 🎯 核心目标

解决当前"伪自动"问题，实现 Claude Code 平台级真正的自动触发。

## 🔍 问题分析

当前 CLAUDE.md 中只有触发规则文字表格，实际流程是：

1. 用户输入"开发前端页面"
2. 主 Agent 读取 CLAUDE.md，看到表格写着应该调用 frontend-developer
3. 主 Agent 自己判断后，主动使用 Agent 工具调用
   → 这不是真正的自动，依赖大模型理解能力

## 📋 任务清单

### 第一阶段：分析与方案

1. 对比分析：当前"伪自动"方案 vs 平台"真自动"方案
2. 输出方案：包含触发机制对比表、设计理由、实施步骤
3. 用户确认方案后再执行

### 第二阶段：命名统一

1. 检查所有 18 个 Agent 的 name 字段与文件名是否一致
2. 修复不一致的命名（如 -agent 后缀问题）

### 第三阶段：添加 triggers 配置

为所有 18 个 Agent 在 YAML frontmatter 中添加 triggers 字段，要求：

- 每个 Agent 至少 4 个触发词，建议 5-10 个
- 覆盖多种表述：动词开头、名词开头、中英文、简称
- 分类配置：前端类、后端类、通用工具类

### 第四阶段：文档完善

1. 在 CLAUDE.md 中补充核心调用机制说明
2. 补全缺失的 Agent 触发规则表格
3. 移除不存在的 Agent 条目

## 📊 输出要求

1. 先输出完整方案和理由，用户确认后再执行
2. 每阶段完成后输出验证结果
3. 最终输出完成总结和统计表
```

# 第 5 章：核心知识点沉淀

## 🎯 知识点 1：什么是"伪自动" vs "真自动"

| 对比项   | 伪自动                 | 真自动                     |
| -------- | ---------------------- | -------------------------- |
| 触发层级 | 主 Agent 逻辑判断      | 平台原生机制               |
| 可靠性   | 依赖大模型理解，可能漏 | 100% 确定性                |
| 响应速度 | 先读文档再判断（慢）   | 直接匹配路由（快 5-10 倍） |
| 维护方式 | 集中在文档表格         | 每个 Agent 自描述          |
| 职责边界 | 主 Agent 兼任调度员    | 职责分离，各司其职         |

**记忆口诀**：平台做的才是真自动，AI 自己判断的都是伪自动。

---

## 🎯 知识点 2：Agent 配置的 YAML frontmatter 结构

标准的 Claude Code Agent 配置结构：

```yaml
---
name: agent-name # 必须与文件名一致
description: 一句话描述 # 简要说明用途
tools: Read, Write, Edit # 可用工具列表
model: inherit # 模型配置（inherit/sonnet/opus/haiku）
triggers: # 自动触发关键词列表
  - 触发词 1
  - 触发词 2
  - ...
color: blue # 可选：显示颜色
---
# Agent 的系统提示内容...
```

**必填字段**：`name`、`description`、`tools`
**推荐字段**：`triggers`（实现自动触发的核心）
**可选字段**：`model`、`color`

---

## 🎯 知识点 3：触发词设计的 4 个维度

为了提高匹配率，触发词设计需要覆盖以下 4 个维度：

| 维度           | 示例                        | 说明                   |
| -------------- | --------------------------- | ---------------------- |
| **动词开头**   | "开发前端页面"、"创建组件"  | 用户最常用的祈使句表述 |
| **名词开头**   | "前端开发"、"代码审查"      | 偏名词性的需求表述     |
| **中英文混合** | "Code Review"、"React 开发" | 技术人员常用的中英混搭 |
| **简称缩写**   | "CR"、"Bug"、"API"          | 简写和行业术语         |

**设计原则**：宁滥勿缺，多几个触发词不会有坏处，少了就匹配不到。

# 第 6 章：踩坑教训与避坑指南

## ⚠️ 踩坑记录 1：工具调用格式检测导致 Write 失败

**现象**：Write 工具写入文档内容时，如果内容中包含工具调用格式的 XML 标签，会被系统安全机制拦截，报 InputValidationError。

**根因**：Claude Code 有安全检测，防止在文档中误触发工具调用。

**解决方案**：

1. 将示例代码中的 XML 标签改为文字描述
2. 或者用代码块包裹，但去掉 `<` 和 `>` 符号
3. 或者用文字说明"参考 Agent 工具的标准调用格式"

**预防措施**：写入文档前，先检查内容中是否有 `<Agent>`、`<file_path>` 等敏感标签。

---

## ⚠️ 踩坑记录 2：命名不一致导致 Agent 调用失败

**现象**：前端 3 个 Agent 的 name 字段带 `-agent` 后缀，但文件名和 CLAUDE.md 表格中不带，导致按照表格中的值调用时找不到 Agent。

**根因**：历史遗留问题，命名规范不统一。

**解决方案**：统一移除 `-agent` 后缀，保持以下三者一致：

1. 文件名（不含 .md）
2. YAML frontmatter 中的 name 字段
3. CLAUDE.md 表格中的 subagent_type 值

**预防措施**：新增 Agent 时，必须检查三者命名一致。

---

## ⚠️ 踩坑记录 3：触发词太少导致匹配率低

**现象**：如果每个 Agent 只有 1-2 个触发词，用户换一种说法就匹配不到了。

**根因**：用户表述方式多样，单一触发词覆盖范围有限。

**解决方案**：每个 Agent 至少配置 4 个触发词，建议 5-10 个，覆盖 4 个维度（动词、名词、中英文、简称）。

---

# 第 7 章：扩展思考与最佳实践

## 💡 最佳实践 1：先方案后执行，永远不要上来就干

- **为什么**：AI 的理解可能和用户预期有偏差，先对齐再动手，避免返工
- **怎么做**：输出 3-5 条核心方案要点 + 理由，用户确认后再开始编码
- **ROI**：多花 2 分钟写方案，少花 20 分钟返工，收益 10 倍以上

---

## 💡 最佳实践 2：配置类任务必须有验证步骤

- **为什么**：配置写错了（如缩进、拼写），当时看不出来，真正用时才发现
- **怎么做**：配置完成后，用脚本或命令批量验证，输出统计表格
- **示例**：本次完成后输出了"18 个 Agent，总计 100 个触发词"的统计

---

## 💡 最佳实践 3：同类任务一定要沉淀模板

- **为什么**：80% 的任务都是重复的，沉淀模板后，下次效率提升 500%
- **怎么做**：把本次的需求整理成标准化模板，包含：目标、问题、任务清单、验收标准
- **收益**：下次做类似任务，复制模板改几个参数就行，不用重新组织语言
