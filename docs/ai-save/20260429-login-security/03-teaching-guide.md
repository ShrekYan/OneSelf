# 第 5 章：知识点与踩坑教训

---

## 💡 核心知识点沉淀

### 知识点 1：前端安全分层策略

**记忆口诀**：Cookie 存 Token，Storage 存缓存，日志要干净。

| 数据类型           | 存储位置                 | 安全要求               |
| ------------------ | ------------------------ | ---------------------- |
| 认证 Token         | HttpOnly Cookie          | 绝对不能让前端 JS 接触 |
| 用户信息（非敏感） | sessionStorage（编码后） | 防肉眼读取即可         |
| 业务配置           | localStorage             | 可以明文，但要防篡改   |
| 调试日志           | 仅开发环境输出           | 生产环境必须删除       |

**反例教训**：很多项目把 JWT 存在 localStorage，XSS 攻击一拿一个准。

---

### 知识点 2：console 日志的安全风险等级

| 日志类型               | 风险等级 | 处理策略                                 |
| ---------------------- | -------- | ---------------------------------------- |
| API 响应（含用户信息） | 🔴 高危  | 生产环境必须删除                         |
| 错误堆栈               | 🟠 中危  | 生产环境仅上报到监控系统，不输出到控制台 |
| 调试信息（跳转、点击） | 🟡 低危  | 生产环境删除                             |
| 错误提示（纯文本）     | 🟢 安全  | 可以保留                                 |

**最佳实践**：通过构建工具（Vite/esbuild/Terser）自动删除生产环境 console，一劳永逸。

```typescript
// vite.config.ts
export default defineConfig({
  esbuild: {
    drop: process.env.NODE_ENV === 'production' ? ['console', 'debugger'] : [],
  },
});
```

---

### 知识点 3：Base64 不是加密，但 90% 场景足够

| 对比项     | Base64 编码          | AES 加密                  |
| ---------- | -------------------- | ------------------------- |
| 依赖       | 零依赖（浏览器原生） | 需要 crypto-js（~40KB）   |
| 性能       | 纳秒级，可忽略       | 毫秒级，有明显开销        |
| 防肉眼读取 | ✅ 可以              | ✅ 可以                   |
| 防专业黑客 | ❌ 不行              | ✅ 可以，但密钥管理是难题 |
| 代码量     | ~20 行               | ~100 行 + 依赖管理        |

**结论**：90% 的前端场景，Base64 就足够了。真正的敏感数据不要存在前端。

## ⚠️ 踩坑教训清单

### 坑 1：只改写入，不改读取

**现象**：登录时把 userInfo 编码后存入 sessionStorage，但其他模块读取时还是直接 JSON.parse，结果全是 null，用户一直登不上。

**根因**：sessionStorage 是全局共享的，写和读必须成对修改。

**预防**：修改前全局搜索，找到所有读写点，列清单后逐个改造。

---

### 坑 2：异常处理缺失

**现象**：遇到非法编码的字符串时，`atob()` 抛出异常，导致整个页面白屏。

**根因**：只考虑了正常路径，没考虑异常路径。

**预防**：所有解码操作都要 try-catch，失败返回 null 或默认值。

```typescript
// ✅ 正确写法
try {
  return JSON.parse(decodeURIComponent(atob(str)));
} catch {
  return null;
}

// ❌ 错误写法（没有异常处理）
return JSON.parse(decodeURIComponent(atob(str)));
```

---

### 坑 3：特殊字符编码问题

**现象**：用户昵称包含 emoji 或中文，直接 btoa 报错。

**根因**：btoa 只支持 Latin-1 字符集，不支持 Unicode。

**预防**：先 encodeURIComponent，再 btoa。

```typescript
// ✅ 正确写法
btoa(encodeURIComponent(JSON.stringify(data)));

// ❌ 错误写法（中文/emoji 会报错）
btoa(JSON.stringify(data));
```

---

### 坑 4：过度设计

**现象**：为了"真正安全"，引入 crypto-js 做 AES 加密，结果密钥硬编码在前端代码里，还是能被找到。

**根因**：没有理解安全的本质，前端不可能安全保存密钥。

**预防**：分层防御，真正敏感的数据不要存在前端。能放在 HttpOnly Cookie 里的，绝对不要让 JS 接触。
