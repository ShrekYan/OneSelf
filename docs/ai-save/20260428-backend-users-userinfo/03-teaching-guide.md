## 一、知识点总结

### 🎯 核心知识点 1：微服务服务边界划分原则

**学习要点**：

| 服务                    | 职责边界                       | 不应该做的事                   |
| ----------------------- | ------------------------------ | ------------------------------ |
| **auth-service**        | 认证、授权、令牌签发、令牌验证 | 查询用户业务信息、处理业务逻辑 |
| **backend（业务服务）** | 业务数据查询、业务逻辑处理     | 自己实现 JWT 解析和验证        |

**架构师思维**：

> 认证是横切关注点，应该统一由认证服务处理。业务服务不应该关心 Token 如何解析，只需要拿到验证后的 userId。这样做的好处是：认证逻辑变更只需要改一个地方，所有业务服务自动受益。

**反面教材**：
每个业务服务都自己实现 JWT 解析，密钥散落在各个服务配置中，一旦要改密钥，所有服务都要改配置重新部署。

---

### 🎯 核心知识点 2：跨服务 DTO 对齐的正确姿势

**学习要点**：

| 方案                 | 做法                               | 优缺点                                                |
| -------------------- | ---------------------------------- | ----------------------------------------------------- |
| **方案 A ✅ 推荐**   | 每个服务独立定义 DTO，保持字段一致 | ✅ 服务解耦、独立部署、变更不互相影响 ❌ 少量代码重复 |
| **方案 B ❌ 不推荐** | 服务 A import 服务 B 的内部类型    | ✅ 代码零重复 ❌ 强耦合、编译依赖、部署绑定           |

**架构师思维**：

> "少量的代码重复"远远好于"不恰当的依赖耦合"。DTO 是接口契约，契约应该稳定且独立，而不是被内部实现牵着走。

**本次实践**：
backend 独立创建 UserDto，与 auth-service 逐字段 100% 对齐，但不 import auth-service 的任何类型。

---

### 🎯 核心知识点 3：接口开发后的深度自检流程

**本次演示的 7 层深度检查法**：

```
第 1 层：编译检查 → npx tsc --noEmit
第 2 层：代码规范 → npm run lint
第 3 层：路由验证 → 全局前缀 + 版本控制 + 控制器路径
第 4 层：数据流验证 → Guard 挂载 userId → 装饰器提取 → Service 使用
第 5 层：返回格式验证 → 全局拦截器是否正确包裹
第 6 层：数据库字段验证 → DTO 字段与 schema 是否匹配
第 7 层：Swagger 文档验证 → 注解是否完整正确
```

**架构师思维**：

> 写完代码只是完成了 50%，剩下的 50% 是证明它确实能在各种条件下正确运行。深度自检不是浪费时间，是减少线上 bug 的最有效手段。

### 🎯 核心知识点 4：TypeScript 类型安全的细节

**本次发现并修复的隐性问题**：

**问题代码**（原代码中存在）：

```typescript
bio: (user as any).bio ?? undefined;
```

**问题分析**：

1. `as any` 绕过了类型检查，失去了 TypeScript 的保护
2. 数据库 Users 表根本没有 `bio` 字段！查询出来永远是 `undefined`
3. 这段代码是无效的死代码，但类型系统没有报警

**修复后**：

```typescript
// 直接移除 bio 字段，数据库本来就没有
```

**架构师思维**：

> `any` 是类型安全的毒品。每多一个 `as any`，类型系统的保护就少一分。当你想写 `as any` 的时候，停下来想想：是不是数据模型本身有问题？

## 四、踩坑教训记录

### 💥 踩坑 1：修改装饰器忘记改导入

**现象**：
把 `@Get('info')` 改成 `@Post('info')` 后，编译报错 `Cannot find name 'Post'`

**原因分析**：
只改了装饰器的使用，忘记检查导入语句是否需要同步变更。

**正确做法**：
修改装饰器后，立刻检查顶部的 import 语句：

```typescript
// 修改前
import { Controller, Get, Put, Body, UseGuards } from '@nestjs/common';

// 修改后（必须同步改）
import { Controller, Post, Put, Body, UseGuards } from '@nestjs/common';
```

**预防措施**：
养成习惯：只要改了装饰器名称，立刻向上滚动到文件顶部检查导入。

---

### 💥 踩坑 2：/save-ai 命令的文档分层问题

**现象**：
第一次执行 /save-ai 时，把"提示词对照学习区"写到了对话回复里，但**没有写入到 03-teaching-guide.md 文件中**，导致用户打开文件发现原始提示词丢失。

**原因分析**：

- /save-ai 命令要求生成 3 个文档，其中第 3 个教学文档**必须包含提示词对照学习区**
- 但我错误地把最高优先级的内容放在了对话回复里，而不是文件里
- 更严重的是：文件中的自我校验还写了"✅ 原始提示词完整保留"，这是虚假校验

**正确做法**：

1. 提示词对照学习区必须写入到 03-teaching-guide.md 的最前面
2. 写入完成后再在对话回复里做总结
3. 自我校验必须真实反映文件的实际内容

**预防措施**：
执行 /save-ai 时，永远先写教学文档文件（包含完整的提示词对照区），文件写入完成后再在对话里回复。

---

### 💥 踩坑 3：DTO 字段与数据库不一致

**现象**：
深度检查时发现 `UpdateProfileDto.bio` 字段，但 Prisma schema 中 Users 表根本没有 bio 字段。

**原因分析**：
历史遗留代码，DTO 定义与数据库 schema 不同步。

**正确做法**：

1. 每次修改 DTO 时，都去对照一下 Prisma schema
2. 写 Service 时，如果发现需要 `as any` 才能访问某个字段，停下来想：数据库里真的有这个字段吗？
3. `as any` 本身就是一个"代码坏味道"的信号

**预防措施**：
`as any` = 红色警报。任何 `as any` 的出现都值得多问一句：为什么需要这个？是不是我的模型哪里错了？

## 五、本次协同的最佳实践总结

### ✅ 最佳实践 1："方案审查 → 代码实现" 两步走

**做法**：

1. 先让 AI 给出完整方案和评估建议
2. 用户审查确认方案无误
3. 再开始写代码

**为什么好**：

- 方向性错误在写代码前就被发现，避免返工
- 用户和 AI 对最终目标达成共识
- 减少"你怎么这么做？我要的不是这个"的沟通成本

---

### ✅ 最佳实践 2：深度自检 > 等用户找 bug

**做法**：
代码写完后，AI 主动进行 7 层深度检查，自己先找问题，确认无问题后再告诉用户。

**为什么好**：

- 用户的时间是宝贵的，不应该让用户当测试员
- AI 自检比人工审查更全面、更快
- 养成好习惯，减少线上 bug

---

### ✅ 最佳实践 3：小步快跑，增量验证

**做法**：

- 不是等所有代码写完再验证
- 每改完一个文件，就跑一次类型检查
- 代码全部写完后，再做整体架构层面的深度检查

**为什么好**：

- 问题发现得越早，修复成本越低
- 避免最后积累一堆问题无从下手
- 每一步都有正向反馈，信心更足
