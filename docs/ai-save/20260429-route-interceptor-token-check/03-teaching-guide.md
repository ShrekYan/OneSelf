## 第 5 章：知识点与踩坑教训

### 🎓 知识点 1：循环依赖的识别与解决

**什么是循环依赖？**

模块 A 导入模块 B，模块 B 又导入模块 A，形成一个闭环。在 ES Module 中，模块导出是活绑定，但初始化顺序不对时，就会出现变量还没赋值就被引用的情况。

**本次的循环依赖链：**

```
axios-instance.ts
    ↓ import defaultApi from '../index'
index.ts
    ↓ import * as authApi from './auth'
auth/index.ts
    ↓ import api from '../core/axios-instance'
axios-instance.ts  ← 回到起点，形成死循环！
```

**解决方法的核心思路：**

打破链条中的任意一环即可。本次选择在 `auth/index.ts` 这一环打破，改为从入口文件导入，因为入口文件的导出已经完成了声明。

**快速识别循环依赖的 3 个特征：**

1. 报错信息包含 `Cannot access 'xxx' before initialization`
2. 报错位置在 `import` 语句或模块顶层
3. 修改某一行导入语句后问题神奇地消失了

### 🎓 知识点 2：认证接口的特殊配置

**为什么 `getCurrentUser()` 需要 `skipAuth: true`？**

这是一个非常经典的"鸡生蛋、蛋生鸡"问题：

```
要调用 getCurrentUser() 验证是否登录
    ↓
API 拦截器发现没有 Token，返回 401
    ↓
401 处理器触发跳转登录页
    ↓
但我们本来就是在验证是否登录，这就形成了死循环！
```

**解决方案**：认证接口本身必须跳过认证拦截，自己处理失败情况。

**这个配置的重要性被严重低估了**：90% 的新手写认证接口时都会踩这个坑，而且这个 Bug 的表现非常诡异（页面无限重定向），排查起来非常耗时。

---

### 🎓 知识点 3：Mock 接口替换的正确流程

**新手常见流程（错误）：**

```
1. 直接把 Mock 函数删了
2. 写个 fetch 调一下后端
3. res.json() 拿数据
4. 能跑就行，收工
```

**架构师标准流程（正确）：**

```
1. ✅ 确认后端接口标准：路径、方法、参数、返回类型
2. ✅ 检查是否已有可复用的接口和类型，不要重复造轮子
3. ✅ 在 API 模块中完善接口，加上必要的配置（skipAuth 等）
4. ✅ 在业务代码中调用统一的 API 函数，不要用原生 fetch
5. ✅ 类型检查 + Lint 检查
6. ✅ 验证异常场景：Token 失效时会不会正确跳转，会不会死循环
```

**两者的本质区别**：新手只关心"成功路径能不能跑"，架构师关心"所有路径是不是符合架构规范"。今天省的 5 分钟，明天就是 5 小时的技术债。
