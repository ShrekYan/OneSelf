# T0 安全漏洞修复方案（分布式架构版）

**文档版本**: v1.0
**创建日期**: 2026-04-28
**适用环境**: 微服务分布式部署
**风险级别**: 🔴 T0 高危 - 立即修复

---

## 一、问题概述

本次修复针对 2 个 T0 高危安全漏洞：

| 编号 | 安全漏洞                          | 风险级别 | OWASP 分类        | 业务影响                             |
| ---- | --------------------------------- | -------- | ----------------- | ------------------------------------ |
| T0-1 | 登录接口无速率限制                | 🟥 高危  | A7 - 安全配置错误 | 攻击者可无限次尝试密码，实施暴力破解 |
| T0-2 | console.log 明文记录 refreshToken | 🟥 高危  | A3 - 敏感数据泄露 | 日志泄露导致用户会话被劫持           |

> **重要说明**：本项目采用**微服务分布式架构**（auth-service + backend-service + log-service），速率限制必须支持**多实例部署场景**，内存存储方案在分布式下无效。

##### 步骤 2：配置 ThrottlerModule（Redis 存储）

```typescript
// services/backend/src/app.module.ts
import { Module } from '@nestjs/common';
import { ThrottlerModule, ThrottlerGuard } from '@nestjs/throttler';
import { ThrottlerStorageRedisService } from 'nestjs-throttler-storage-redis';
import Redis from 'ioredis';
import { APP_GUARD } from '@nestjs/core';
import { CustomThrottlerGuard } from './common/guards/throttler.guard';

@Module({
  imports: [
    // ... 其他模块

    // 速率限制模块 - Redis 集中式存储
    ThrottlerModule.forRoot({
      throttlers: [
        {
          ttl: 60, // 时间窗口：60 秒
          limit: 30, // 全局默认限制：30 次
        },
      ],
      ignoreUserAgents: [/googlebot/gi, /bingbot/gi], // 搜索引擎爬虫豁免
      storage: new ThrottlerStorageRedisService(
        new Redis({
          host: process.env.REDIS_HOST || 'localhost',
          port: parseInt(process.env.REDIS_PORT || '6379'),
          password: process.env.REDIS_PASSWORD,
          db: parseInt(process.env.REDIS_THROTTLE_DB || '1'), // 使用独立 DB
          keyPrefix: `${process.env.NODE_ENV || 'dev'}:throttle:`, // 环境隔离
          enableReadyCheck: true,
          maxRetriesPerRequest: 1, // Redis 故障时快速降级
          enableOfflineQueue: false,
        }),
      ),
    }),
  ],
  providers: [
    // ... 其他 Provider

    // 全局速率限制 Guard
    {
      provide: APP_GUARD,
      useClass: CustomThrottlerGuard,
    },
  ],
})
export class AppModule {}
```

##### 步骤 3：自定义 Guard（支持白名单 + 真实 IP）

```typescript
// services/backend/src/common/guards/throttler.guard.ts
import { Injectable, ExecutionContext } from '@nestjs/common';
import { ThrottlerGuard } from '@nestjs/throttler';
import { Request } from 'express';

@Injectable()
export class CustomThrottlerGuard extends ThrottlerGuard {
  /**
   * 获取客户端真实 IP（支持 Nginx 反向代理）
   */
  protected override async getTracker(req: Request): Promise<string> {
    const xRealIp = req.headers['x-real-ip'] as string;
    if (xRealIp) return xRealIp;

    const xForwardedFor = req.headers['x-forwarded-for'] as string;
    if (xForwardedFor) {
      return xForwardedFor.split(',')[0].trim();
    }

    return req.ip || 'unknown';
  }

  /**
   * 判断是否跳过限流
   */
  protected override async shouldSkip(context: ExecutionContext): Promise<boolean> {
    // 功能开关：可通过环境变量临时关闭限流
    if (process.env.THROTTLE_ENABLED === 'false') {
      return true;
    }

    const req = context.switchToHttp().getRequest<Request>();

    // 1. 健康检查接口跳过限流
    if (req.url.includes('/health') || req.url.includes('/actuator')) {
      return true;
    }

    // 2. 内部服务 IP 白名单
    const internalIps = (process.env.INTERNAL_SERVICE_IPS || '')
      .split(',')
      .map(ip => ip.trim())
      .filter(ip => ip);

    const clientIp = await this.getTracker(req);
    if (internalIps.includes(clientIp)) {
      return true;
    }

    return false;
  }

  /**
   * Redis 故障降级处理
   * 重写 throwIf 方法，捕获异常后跳过限流
   */
  protected override async throwIf(
    context: ExecutionContext,
    limit: number,
    ttl: number,
    tracker: string,
  ): Promise<void> {
    try {
      await super.throwIf(context, limit, ttl, tracker);
    } catch (error) {
      // 如果是 Redis 连接错误，记录告警后跳过限流
      if (error.message.includes('Redis') || error.message.includes('ECONN')) {
        // 这里可以触发告警（如发送到 log-service）
        console.warn('[THROTTLE DEGRADED] Redis unavailable, skipping rate limit:', error.message);
        return;
      }
      // 正常的 429 限流错误继续抛出
      throw error;
    }
  }
}
```

## 四、限流阈值设计说明

| 接口                       | 阈值          | 设计考虑                                |
| -------------------------- | ------------- | --------------------------------------- |
| `/api/auth/login`          | **5次/60秒**  | 防止暴力破解，正常用户不会频繁登录      |
| `/api/auth/register`       | **3次/300秒** | 防止垃圾账号注册，降低短信/邮件轰炸风险 |
| `/api/auth/password-reset` | **2次/600秒** | 防止密码重置接口被滥用                  |
| **全局默认**               | **30次/60秒** | 保护普通业务接口，防止爬虫和 DoS 攻击   |

> **调整策略**：上线初期阈值可适当放宽，观察 1-2 周后逐步收紧到推荐值。

## 五、分布式架构验证清单

| 验证场景           | 验证方法                                                   | 预期结果                          |
| ------------------ | ---------------------------------------------------------- | --------------------------------- |
| **单实例限流**     | 单实例连续调用 login 接口 6 次                             | 第 6 次返回 429 Too Many Requests |
| **多实例共享计数** | 启动 2 个 backend 实例<br>实例1 调用 3 次，实例2 调用 3 次 | 合计第 6 次返回 429               |
| **不同 IP 隔离**   | IP A 调用 5 次被限制，IP B 继续调用                        | IP B 正常响应，计数独立           |
| **内部服务白名单** | 从 auth-service IP 调用 login 10 次                        | 不触发限流，正常响应              |
| **健康检查豁免**   | 频繁调用 /health 接口（每秒 1 次）                         | 不触发限流                        |
| **Redis 故障降级** | 关闭 Redis 后正常调用接口                                  | 自动降级跳过限流，不影响业务      |
| **Token 不泄露**   | 执行登出操作后 grep 所有日志                               | 无 refreshToken 明文出现          |
| **正常流程验证**   | 完整登录 → 业务操作 → 登出流程                             | 功能完全正常，无报错              |

## 十一、后续优化方向

1. **用户级限流**：登录后按 User ID 限流，不局限于 IP
2. **动态阈值**：基于机器学习自动调整限流策略
3. **验证码触发**：触发限流后要求输入验证码，而不是直接拒绝
4. **黑名单机制**：自动封禁频繁触发限流的恶意 IP
