# 🎯 核心优化领域与检查要点

## 1. React 组件优化

### 必须检查

- **不必要重渲染**：识别哪些组件在不需要更新时重复渲染
- **组件拆分**：过大组件需要拆分为更小的组件，实现精准更新
- **`useCallback`/`useMemo` 合理使用**：避免过度使用也避免遗漏
- **React 19 特性**：
  - ✅ 利用 `useTransition` 标记非紧急更新，避免阻塞渲染
  - ✅ 利用 `useDeferredValue` 延迟非紧急值的更新
  - ✅ 利用 React Compiler 自动优化（项目已启用）
- **懒加载**：非首屏组件使用 `React.lazy()` 懒加载
- **虚拟列表**：长列表必须使用虚拟滚动，避免渲染上千 DOM 节点

### 优化检查清单

- [ ] 所有组件 `props` 是否保持引用稳定？
- [ ] 是否存在将对象/函数字面量直接传递给组件的情况？
- [ ] 列表渲染是否使用了稳定的 `key` 属性？
- [ ] 是否有可以按需懒加载的组件？
- [ ] 长列表是否使用了虚拟化？

## 2. MobX 状态管理优化

### 必须检查

- **可观察粒度**：是否存在不必要的大对象整体观察？
- **`useObserver` 作用范围**：是否只在真正需要观察的组件使用？
- **派生数据**：计算属性是否使用 `computed` 缓存？
- **批量更新**：多次状态更新是否使用 `runInAction` 批量处理？
- **页面级状态**：页面退出后是否清理了状态？

### 常见问题

❌ 错误做法：
```tsx
// 整个大对象被观察，任何字段变化都会导致所有订阅者重渲染
const store = useLocalObservable(() => ({
  bigData: { ... } // 包含很多不相关字段
}));
```

✅ 正确做法：
```tsx
// 拆分独立状态，只有用到的变化才会触发重渲染
const store = useLocalObservable(() => ({
  data1: ...,
  data2: ...,
  get filteredData() {
    // computed 缓存结果
    return this.data1.filter(...);
  }
}));
```

## 3. 移动端 H5 性能最佳实践

### 内存优化

- **图片懒加载**：非首屏图片必须懒加载，避免一次性加载大量图片
- **事件监清理**：`useEffect` 清理定时器、事件监听器、订阅
- **避免内存泄漏**：检查全局缓存、定时器、未清理的事件监听
- **路由缓存策略**：使用 `react-activation` + `react-freeze` 合理缓存页面，避免缓存过多页面

### 交互性能

- **触摸区域**：可点击元素尺寸 ≥ 44px × 44px（项目规范强制要求）
- **点击反馈**：按钮需要有 `:active` 状态反馈
- **防抖节流**：频繁触发的事件（scroll、resize、input）使用防抖节流
- **避免强制同步布局**：读写 DOM 操作分离，避免连续强制重排
- **安全区域适配**：适配刘海屏、底部指示条等安全区域

### 加载性能

- **图片优化**：
  - 优先使用 WebP 格式
  - 适当压缩，避免大图小用
  - 使用 `srcset` 适配不同设备像素比
- **CSS 优化**：
  - 避免嵌套过深（≤ 3 层，项目规范强制要求）
  - 避免昂贵 CSS 属性（如 `box-shadow`、`filter` 过度使用）
- **JavaScript 打包优化**：
  - 路由级代码分割
  - 第三方库分离打包
  - 检查打包体积，识别过大依赖

## 4. 网络请求优化

### 必须检查（基于项目现有 API 架构）

项目 API 层已经内置以下优化，检查是否正确使用：

- ✅ **重复请求取消**：`cancel-manager` 自动取消重复并发请求
- ✅ **请求缓存**：不变数据（如分类列表、配置信息）启用 `cache: true`
- ✅ **智能重试**：网络抖动自动重试

### 优化建议

- **批量请求**：避免多次小请求，能合并则合并
- **懒加载数据**：需要时再请求数据，不要一次性预加载所有
- **分页加载**：列表数据分页加载，避免一次性加载全部
- **数据缓存**：不常变数据合理缓存，减少重复请求
- **错误重试**：幂等请求配置合理重试次数

## 5. 内存泄漏检测

### 常见泄漏点检查

1. **定时器/计时器**：`setInterval`/`setTimeout` 未清理
2. **事件监听器**：`addEventListener` 未 `removeEventListener`
3. **全局缓存**：`localStorage`/`Map` 持续增长不清理
4. **订阅模式**：MobX 观察者、第三方库订阅未取消订阅
5. **闭包引用**：闭包持有大对象引用无法释放
6. **DOM 引用**：已经移除的 DOM 节点仍被引用

### 检测方法

- 使用 Chrome DevTools Memory 标签页拍堆快照
- 检查 Performance 记录中内存是否持续增长
- 多次路由切换后内存是否回落

## 6. Lighthouse 审计维度

按照 Lighthouse 六大性能维度系统性检查优化：

### 1. 性能 (Performance)
- **First Contentful Paint (FCP)** → 优化资源加载，压缩打包
- **Largest Contentful Paint (LCP)** → 优化最大内容元素加载，预加载关键资源
- **Cumulative Layout Shift (CLS)** → 图片设置宽高比，避免布局偏移
- **First Input Delay (FID)** / **Interaction to Next Paint (INP)** → 减少长任务，拆分大块计算

### 2. 可访问性 (Accessibility)
- 图片有 `alt` 属性
- 可点击元素有足够点击区域
- 颜色对比度符合标准

### 3. 最佳实践 (Best Practices)
- 避免滥用第三方广告脚本
- 使用 HTTPS
- 避免已知安全问题的依赖

### 4. SEO
- 移动端 viewport 配置正确
- 页面有合理标题描述
- 链接可爬取

# 🔍 常用优化技巧速查

| 场景 | 优化方案 |
|---|---|
| 组件频繁重渲染 | 使用 `useCallback` 稳定引用，`useMemo` 缓存计算结果 |
| 大列表渲染 | 使用虚拟滚动，只渲染可见区域 |
| 不常变数据 | API 层启用 `cache: true` 缓存 |
| 非紧急更新 | 使用 `useTransition` 让出优先级 |
| 重复并发请求 | 依赖项目 `cancel-manager` 自动取消 |
| 路由切换后状态 | 不需要缓存的及时清理，需要缓存的使用 `react-activation` |
| 图片展示 | 懒加载 + WebP + 合理压缩 |
| 大量图片 | 使用渐进式加载，降低初始内存占用 |
| 频繁事件 | 使用 `debounce`/`throttle` 限制触发频率 |
