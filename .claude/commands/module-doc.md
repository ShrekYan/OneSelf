# 通用模块文档生成命令

**触发**: `/module-doc [可选：模块路径]`
- `模块路径`（可选）：手动指定模块路径，覆盖自动检测

**行为**: 智能分析对话内容，自动识别开发类型（前端/后端/全栈/数据库等），动态注入对应侧重点的检查清单和最佳实践，生成结构化的模块说明文档。

---

## 核心设计理念

**一个通用模板 + N 个内容配置 = 无限适配能力**

- ✅ 统一模板结构，消除重复维护
- ✅ 智能类型识别，自动注入对应内容
- ✅ 支持多类型组合，全栈开发同时注入前后端内容
- ✅ 新增类型零成本，只需要加配置

---

## 执行流程

### 第一步：智能类型识别

自动分析当前对话内容和修改文件，命中 2 个以上关键词即匹配对应类型。

#### 🔍 类型配置清单

| 类型 | 关键词 | 配置 ID |
|------|--------|---------|
| **前端 React 组件** | React, useState, useEffect, useMemo, useCallback, MobX, observer, useLocalObservable, 组件, 页面, 前端, Ant Design Mobile, CSS, SCSS, 样式 | `frontend-react` |
| **后端 NestJS 模块** | NestJS, Controller, Service, Module, DTO, 中间件, 拦截器, 守卫, 后端, 接口, API | `backend-nestjs` |
| **数据库 Prisma** | Prisma, schema, model, migrate, 数据库, MySQL, PostgreSQL, Redis, SQL, 索引, 事务 | `backend-prisma` |
| **工具函数/通用** | utils, helper, 工具函数, 公共方法, 通用逻辑 | `common-utils` |

> 💡 **支持多类型匹配**：同时命中多个类型时，自动组合注入所有匹配类型的内容。

---

### 第二步：动态内容注入

根据识别到的类型，从配置中注入对应内容。

#### 📦 注入内容结构

每个类型配置包含 4 个注入点：

```typescript
{
  typeId: string,           // 类型 ID
  displayName: string,      // 显示名称（如：前端 React 组件）
  checklists: string[],     // 质量检查清单项
  examples: string[],       // 代码示例
  bestPractices: string[],  // 最佳实践
  cautions: string[],       // 注意事项
}
```

## 🎨 各类型动态注入内容

### 类型 1：前端 React 组件 (`frontend-react`)

#### 注入章节：React 组件设计检查清单

```markdown
---

## 🎨 React 组件设计检查清单

- ✅ Props 类型定义是否完整，是否有必要的默认值
- ✅ 是否处理了 Loading 和 Error 状态
- ✅ 样式是否使用 CSS Modules，根容器命名是否规范 `xxxContainer`
- ✅ 组件是否符合单一职责，有没有过大的组件需要拆分
- ✅ Hooks 使用是否正确，有没有违反 Hooks 规则
- ✅ 有没有不必要的重渲染，是否需要 useMemo/useCallback
- ✅ 有没有正确清理 useEffect 的副作用
- ✅ 移动端点击区域是否 ≥ 44px（750 设计稿 ≥ 88px）
- ✅ 可点击元素是否有 :active 反馈效果
- ✅ 是否适配了底部安全区域 `env(safe-area-inset-bottom)`
- ✅ 是否复用了已有组件，有没有重复造轮子
- ✅ 图片是否处理了懒加载和占位图
```

---

### 类型 2：后端 NestJS 模块 (`backend-nestjs`)

#### 注入章节：NestJS 架构检查清单

```markdown
---

## 🏗️ NestJS 架构检查清单

- ✅ 分层是否合理：Controller / Service / Repository 职责是否清晰
- ✅ Controller 有没有写业务逻辑（业务逻辑应该放 Service）
- ✅ Module 划分是否合理，有没有循环依赖
- ✅ DTO 是否有完整的 class-validator 校验规则
- ✅ 是否正确使用了依赖注入，有没有硬编码依赖
- ✅ 中间件/拦截器/守卫使用是否正确
- ✅ 接口权限控制是否完整，有没有越权访问的可能
- ✅ 异常处理是否统一，有没有随意返回错误信息
- ✅ 是否有接口版本管理
- ✅ 日志记录是否完整，关键操作有没有埋点
```

---

### 类型 3：数据库 Prisma (`backend-prisma`)

#### 注入章节：数据库设计检查清单

```markdown
---

## 🗄️ 数据库设计检查清单

- ✅ Model 命名是否规范（PascalCase 首字母大写）
- ✅ 字段类型和长度是否合适，有没有溢出风险
- ✅ 有没有正确的索引，查询高频字段有没有加索引
- ✅ 关联关系设计是否合理，有没有过度设计
- ✅ 有没有 N+1 查询问题，Prisma include 使用是否正确
- ✅ 事务使用是否正确，有没有遗漏的事务场景
- ✅ 有没有软删除标记，删除操作是真删还是软删
- ✅ 迁移脚本是否可回滚
- ✅ 敏感字段有没有加密处理（密码、Token 等）
```


## 多类型组合示例

### 场景：全栈开发，同时命中 frontend-react + backend-nestjs

输出的文档会自动包含两个类型的章节，自动去重：

```markdown
## 🎨 React 组件设计检查清单
...（前端内容）

## 🏗️ NestJS 架构检查清单
...（后端内容）
```

---

## 特殊情况处理

### 情况 1：无法识别类型
- 只生成通用章节
- 在文档顶部标注：「未能识别到特定开发类型，已生成通用文档」
- 不报错，优雅降级

### 情况 2：用户手动指定模块路径
- 优先使用用户传入的路径
- 跳过自动路径提取步骤
- 根据路径特征辅助判断类型（路径有 `apps/web` 优先前端，有 `services/backend` 优先后端）

### 情况 3：多个模块路径
- 按路径分别生成文档，或者合并成一个文档
- 根据文件数量和关联性自动判断
