# 教学教材生成命令

**触发**: `/teaching-summary`

**行为**: 自动分析当前对话中的开发案例，将其总结成结构化的教学教材，输出到 `/docs` 目录。

---

## 执行流程

1. 分析当前对话中的开发案例（需求描述、问题、修复过程、决策点）
2. 按照 10 部分结构化框架整理教材
3. 输出到 `/docs/[案例名称]-teaching-guide.md` 文件（不能跳过此步骤）
4. 展示生成结果供用户预览

---

## 教材结构框架（10 个部分）

### 第 0 部分：提示词预检查清单
- 7 项标准化检查（现象描述、预期行为、功能完整性、边界值、安全、性能、验收标准）
- 预检查得分机制（满分 7 分）
- 给出补充建议（得分 < 5 分时提示需要补充信息）

### 第 1 部分：用户的决策与提示词分析
- 1.1 原始提示词原文（完整保留用户的需求描述）
- 1.2 这个提示词好在哪里？（7 维度标准化评判）
  - 📝 现象描述、🎯 预期行为、✅ 验收标准
  - 🧩 功能完整性、🔲 边界值处理、🔒 安全考虑、⚡ 性能影响
  - 每个维度：正面教材 vs 反面教材 + 架构师思维解释
- 1.3 用户的关键决策点（5 个架构决策及其正确性分析）
  - 决策点 + 用户的选择 + 正确性分析 + 架构师思维
- 1.4 如果用户没说这些，会发生什么？（反事实推演）
  - 表格："如果没说" → "AI 会怎么做" → "后果" → "需求价值"
- 1.5 需求提示词的不足点与改进建议（客观分析可优化点）

### 第 2 部分：问题全景图
- 整体架构图 / 流程图
- 问题之间的关联关系

### 第 3-5 部分：每个问题的详细分析（每个问题单独一章）
- 根因分析流程图
- 修复方案对比（6 维度量化打分制）
  - 🧩 功能完整性、🔲 边界值处理、🔒 安全考虑、⚡ 性能影响、🔧 可维护性、💰 开发成本
  - 每个维度 1-5 星量化打分
  - 自动计算加权总分（根据场景类型：默认/安全敏感/性能敏感/复杂业务）
  - 🚫 否决项机制（安全≤2星 或 性能≤2星 直接否决）
  - 最终方案选择理由
- 代码实现（修改前 vs 修改后对比）
- 💡 架构师思维：标注每一处为什么这样改是正确的

### 第 6 部分：完整修改清单
- 表格形式：文件路径 + 修改内容 + 修复的问题 + 代码行数

### 第 7 部分：验证方案与测试用例
- 表格形式：测试场景 + 前置条件 + 操作步骤 + 预期结果

### 第 8 部分：教学总结
- 8.1 学到的核心原则（3-5 个）
- 8.2 常见踩坑点总结
- 8.3 问题复盘清单（标准化流程）
  - 5 个阶段 × 4 个检查项 = 20 项完整清单（问题识别 → 根因定位 → 方案选择 → 验证测试 → 预防措施）
  - 本次案例复盘完成情况打分（自动根据对话内容评估）
  - 改进方向建议

### 第 9 部分：实战工具包
- 9.1 Bug 修复检查清单（可复用）
- 9.2 架构决策检查清单
- 9.3 需求提示词可复用模板（学员可以直接套用）

## 教学强化要求

1. **每一处关键决策都要标注**：
   - 🎯 **架构师思维**：解释背后的设计原则
   - ✅ **为什么这是正确的？**：分析决策的正确性

2. **视觉层次要求**：
   - 大量使用表格进行对比
   - 使用 emoji 颜色标注关键信息（✅❌⭐🎯🔍⚖️🔒💡）
   - 使用代码块展示"修改前 vs 修改后"对比
   - 使用流程图/架构图可视化问题

3. **提示词优化案例**：
   - 在 1.5 节中，针对本次提示词的不足点，给出"优化前 vs 优化后"的对比案例
   - 说明每个优化点带来的价值

4. **问题复盘清单生成要求**：
   - 必须生成完整的 5 阶段 20 项检查清单（问题识别、根因定位、方案选择、验证测试、预防措施）
   - 根据本次对话内容，对 20 个检查项逐一评估完成情况（✅ 完成 / ⚠️ 部分完成 / ❌ 未涉及）
   - 自动计算复盘得分（满分 100，每项 5 分）
   - 根据得分情况给出具体的改进方向建议（不要泛泛而谈，要针对本次案例）
   - 用 emoji 标注每个阶段的状态（🔍🧠⚖️✅🛡️）

5. **提示词 7 维度评判标准**：
   - 必须按照固定 7 维度进行评判，不能随意增减维度
   - 每个维度必须给出：正面案例（用户写了什么）、反面案例（糟糕的写法是什么样的）、架构师思维解释
   - **打分标准**（每项 5 分）：
     - ⭐⭐⭐⭐⭐：明确、具体、可执行
     - ⭐⭐⭐⭐：有提到，但不够具体
     - ⭐⭐⭐：有这个意识，但表述模糊
     - ⭐⭐：几乎没提
     - ⭐：完全没涉及

6. **方案对比 6 维度打分规则**：
   - 必须使用 6 维度量化打分，每个维度 1-5 星
   - **每个维度必须有明确的打分依据**（不能只给星不给理由）
   - **维度打分标准**：
     - 🧩 功能完整性：⭐⭐⭐⭐⭐ = 覆盖所有场景，⭐⭐ = 主流程都有遗漏
     - 🔲 边界值处理：⭐⭐⭐⭐⭐ = 所有边界都考虑到，⭐⭐ = 只考虑正常流程
     - 🔒 安全考虑：⭐⭐⭐⭐⭐ = 没有安全隐患，⭐⭐ = 有明显漏洞
     - ⚡ 性能影响：⭐⭐⭐⭐⭐ = 零性能损耗，⭐⭐ = 会导致明显性能问题
     - 🔧 可维护性：⭐⭐⭐⭐⭐ = 代码清晰易改，⭐⭐ = 难以理解和维护
     - 💰 开发成本：⭐⭐⭐⭐⭐ = 1 小时内能完成，⭐⭐ = 需要 1 天以上
   - **权重机制**：根据场景自动选择权重
     - 默认场景：功能 30%、边界 20%、安全 30%、性能 20%
     - 安全敏感场景（登录/支付/权限）：安全 50%
     - 性能敏感场景（列表/首页）：性能 40%
     - 复杂业务场景（表单/工作流）：功能 40%、边界 30%
   - **🚫 否决项**：任何方案满足以下任一条件，直接否决，不看总分
     - 安全维度 ≤ 2 星（有明显安全漏洞）
     - 性能维度 ≤ 2 星（会导致明显性能问题）
     - 功能完整性 ≤ 2 星（核心功能都没覆盖）
     - 需要修改 10+ 个文件（侵入性过大）

7. **场景自动识别规则**：
   - 根据提示词内容自动识别场景类型，选择对应的权重
   - 识别关键词："登录"、"支付"、"权限" → 安全敏感场景；"列表"、"首页"、"加载" → 性能敏感场景
   - 无法识别时使用默认权重

8. **需求提示词不足点分类与案例模板规则**：
   - 必须将不足点按标准化分类列出，不能泛泛而谈
   - **分类标准**（6 大类）：
     1. 📝 **现象描述类**：问题描述不清晰、缺少复现路径
     2. 🎯 **预期行为类**：没有说明正确的行为应该是什么
     3. 🔲 **边界场景类**：没有考虑异常情况、边缘数据、极端场景
     4. 🔒⚡ **安全性能类**：没有提到安全要求或性能约束
     5. ✅ **验收标准类**：没有说明怎么判断完成了
     6. 🤝 **人机协作类**：没有明确交互模式（先审查还是直接执行）
   - 每个不足点必须给出：**优化前原文** + **优化后改写** + **价值说明**
   - 必须用表格形式展示，让用户可以直接对照学习和套用
