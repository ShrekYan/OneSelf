# AI 协同结果智能保存命令

**触发**: `/save-ai [可选：模块路径]`
- `模块路径`（可选）：手动指定本次协同开发的模块路径
- 例如：`/save-ai apps/web/src/pages/ArticleDetail`

**核心定位**: 在人机协同对话过程中，即时保存本次 AI 协助开发的所有成果。**不依赖 Git 变更检测**，而是基于当前对话上下文进行智能提取和归档。

**🚨 强制执行规则：每次执行 /save-ai 必须生成 3 个文档 + 1 个强制章节，缺一不可**

1. ✅ **模块功能文档** - `01-README.md`，模块说明，给后续开发看
2. ✅ **协同成果文档** - `02-collaboration.md`，完整的协同过程记录
   - 🚨 **必须包含「## 🔧 核心代码实现」章节**，不得为空，不得遗漏所有重要代码变更
3. ✅ **教学总结文档** - `03-teaching-guide.md`，知识点和踩坑教训沉淀

---

## 设计理念

### 为什么不依赖 Git？
1. **实时性**：对话过程中随时保存，不需要等代码写完才提交 Git
2. **完整性**：保存的是"协同过程"，不仅是代码变更，还包括思路、决策、踩坑记录
3. **灵活性**：支持中途多次保存，支持实验性代码（不一定提交到 Git）

### 保存的是什么？
本次人机协同对话中产生的所有开发产出，包括：
- ✅ 新创建/修改的代码文件内容
- ✅ 代码审查意见和改进建议
- ✅ 架构设计决策过程
- ✅ 遇到的问题和解决方案
- ✅ AI 给出的解释和思路说明
- ✅ 用户的关键决策和反馈

## 执行流程

### 第一步：上下文分析（自动执行）
1. 扫描当前对话历史，识别本次协同的核心内容
2. 自动提取关键信息：
   - 开发的模块/功能名称
   - 修改/创建的文件路径清单
   - 遇到的技术问题和解决方案
   - 重要的架构决策和理由
   - 代码审查的关键点

### 第二步：模块路径识别（自动 + 手动）
1. **优先使用用户传入的模块路径**（如果提供）
2. **自动识别**：从对话内容中提取提到的文件路径，推断模块路径
   - 前端：识别 `apps/web/src/pages/xxx/` 或 `apps/web/src/components/xxx/`
   - 后端：识别 `services/[service]/src/[module]/`
3. **无法识别时**：提示用户手动指定，或使用默认路径（按时间戳命名）

### 第三步：生成模块功能文档（强制）
- **强制自动调用** `/module-doc` 生成模块功能说明文档
- **必须生成**，不得跳过
- 根据对话上下文智能识别开发类型，动态注入对应检查清单和最佳实践
- 文档位置：`[模块路径]/README.md`

**智能类型识别规则**：命中 2 个以上关键词即匹配对应类型
- 前端 React 组件：React、useState、useEffect、useMemo、useCallback、MobX、observer、useLocalObservable、组件、页面、CSS、SCSS、样式、Ant Design Mobile
- 后端 NestJS 模块：NestJS、Controller、Service、Module、DTO、中间件、拦截器、守卫、后端、接口、API
- 数据库 Prisma：Prisma、schema、model、migrate、数据库、MySQL、PostgreSQL、Redis、SQL、索引、事务
- 工具函数/通用：utils、helper、工具函数、公共方法、通用逻辑

## 特殊情况处理

### 情况 1：用户手动指定模块路径
- 优先使用用户传入的路径参数
- 跳过自动识别步骤
- 直接基于对话上下文生成该模块的协同成果文档

### 情况 2：同一次对话多次执行 /save-ai
- 检测是否已有本次对话的协同成果文档
- 采用增量更新方式，补充新的内容，而不是完全覆盖
- 在文件名中添加版本号或时间戳区分

### 情况 3：跨多个模块的协同
- 自动识别所有涉及的模块
- 分别为每个模块生成独立的协同成果文档
- 在汇总输出中列出所有生成的文件

## 文档存储规范

### 🎯 统一目录设计（所有文档放在同一目录，便于查找）

**所有 3 个文档统一放在同一个协同目录下，不再分散存放**

```
docs/
└── ai-save/                          # /save-ai 命令统一根目录
    └── [YYYYMMDD]-[模块名]/          # 单次协同目录，同一天同模块多次保存加 v2/v3
        ├── 01-README.md              # ✅ 模块功能文档（原 module-doc 产出）
        ├── 02-collaboration.md       # ✅ 协同成果文档（完整过程记录）
        └── 03-teaching-guide.md      # ✅ 教学总结文档（知识点沉淀）
```

**示例**：
```
docs/
└── ai-save/
    ├── 20260427-login-performance/
    │   ├── 01-README.md
    │   ├── 02-collaboration.md
    │   └── 03-teaching-guide.md
    └── 20260428-article-detail-v2/
        ├── 01-README.md
        ├── 02-collaboration.md
        └── 03-teaching-guide.md
```

---

### 📋 目录命名规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **基础格式** | `[YYYYMMDD]-[模块名]` | `20260427-login-performance` |
| **多次保存** | 同一天同模块，末尾加 `v2`/`v3` | `20260427-login-performance-v2` |
| **模块名** | kebab-case 小写，用 `-` 分隔 | `user-profile`, `articles-module` |
| **类型前缀** | 文件名加 `01-`/`02-`/`03-` 前缀，按序号排序，一眼看清顺序 | |

---

### ✅ 统一目录的优势

1. **查找方便**：一次协同的所有产出在同一个文件夹
2. **结构清晰**：序号前缀让文档功能一目了然
3. **版本管理**：同模块多次保存不会覆盖，自动版本化
4. **归档简单**：直接打包整个目录即可备份

---

### ❌ 旧存储位置（已废弃）

以下分散存储方式不再使用：
- ~~`apps/web/src/pages/xxx/README.md`~~ → 不再放在代码目录
- ~~`docs/ai-collaboration/xxx.md`~~ → 不再单独存放
- ~~`docs/ai-teaching/[YYYYMM]/xxx.md`~~ → 不再按月份分目录

## 与其他命令的关系

| 命令 | 职责 | 调用关系 |
|------|------|----------|
| `/save-ai` | 保存人机协同的完整过程、决策、问题、代码变更的综合记录 | 自动调用 `/module-doc` + **必须调用** `/teaching-summary` |
| `/module-doc` | ✨ **通用模块文档生成器（唯一模板），智能识别类型动态注入内容 | 被 `/save-ai` 调用，也可独立使用 |
| `/teaching-summary` | 专门生成教学素材，聚焦知识点和踩坑教训 | 被 `/save-ai` **强制调用 |

---

## 质量保证要求

1. **零侵入**：不修改任何现有代码文件，只生成文档
2. **完整性**：确保不遗漏对话中的关键决策和解决方案
3. **结构化**：文档格式统一，便于后续检索和回顾
4. **可追溯**：文档中包含日期、模块路径、文件清单等元信息
