# 功能模块概览生成规范

当需要理解他人开发的特定功能模块时，按照本规范自动分析该模块，生成详细的模块导览报告，帮助快速上手。

**使用方式**: `/module-overview <module-path>`，例如:
- `/module-overview src/pages/ArticleDetail`
- `/module-overview backend/src/auth`

**使用场景**:
- 接手他人维护的功能，需要快速理解
- 跨团队协作，需要快速熟悉陌生模块
- 代码评审前，需要整体了解模块结构

---

## 核心原则

1. **先识别技术栈**: 根据路径自动识别是前端还是后端模块
   - `src/` → 前端模块（React + TypeScript + MobX）
   - `backend/` → 后端模块（NestJS + TypeScript + Prisma）
2. **由粗到细**: 先说整体位置职责，再讲细节逻辑
3. **突出流向**: 理清数据从哪里来到哪里去
4. **提示踩坑**: 总结容易踩错的地方
5. **指路导航**: 告诉阅读者从哪里开始看代码

---

## 分析步骤

### 第一步：自动识别技术栈

| 路径前缀 | 技术栈 | 分析框架 |
|----------|--------|----------|
| `src/` | 前端（React + MobX H5）| 按组件/Store/Handle 分层分析 |
| `backend/` | 后端（NestJS）| 按 Controller/Service/DTO 分层分析 |

### 第二步：收集信息

1. 读取模块目录下所有关键文件
2. 识别模块入口文件
3. 梳理文件结构和职责分工
4. 提取关键类型/接口定义
5. 追踪依赖关系（内部依赖 + 第三方依赖）
6. 梳理核心业务流程和数据流向

### 第三步：生成报告

按照输出模板整理，针对不同技术栈调整分析重点。

## 针对不同技术栈的分析重点

### 前端模块（React + MobX）重点：

| 关注点 | 分析内容 |
|--------|----------|
| 组件结构 | 是否合理拆分了子组件？ |
| 状态管理 | MobX Store 定义了哪些状态和 Action？ |
| 逻辑拆分 | 是否遵循 `index.tsx (UI) + useStore.ts (状态) + handle.ts (纯逻辑)`？ |
| 类型安全 | Props/API 是否有完整类型定义？ |

### 后端模块（NestJS）重点：

| 关注点 | 分析内容 |
|--------|----------|
| 分层结构 | Controller/Service/DTO 职责是否清晰？ |
| 业务逻辑 | Service 层实现了哪些核心业务规则？ |
| 数据模型 | 数据库模型关系是怎样的？ |
| 异常处理 | 是否有统一的异常处理？ |

---

- ❌ 不要逐行翻译代码
- ❌ 不要重复代码已经说清楚的内容
- ❌ 不要省略关键依赖和数据流向
- ❌ 不要隐瞒潜在问题，坦诚提示风险
