# 重构建议规范

当需要分析现有代码并给出重构建议时，按照本规范进行分析和输出。

---

## 重构分析步骤

### 第一步：代码嗅探（Code Smell 识别）

检查以下常见问题：

1. **冗长** - 函数/组件太长，单个文件超过 500 行
2. **重复** - 复制粘贴的代码，逻辑重复
3. **复杂** - 圈复杂度过高，分支太多
4. **模糊** - 命名不清晰，不知道代码做什么
5. **冗余** - 死代码、注释掉的代码、未使用的导入
6. **依赖过多** - 一个文件依赖太多外部模块
7. **不合理的职责** - 一个组件/函数做太多不相关的事

### 第二步：评估影响

判断问题的严重程度：

- **严重**: 影响功能、导致 bug、难以维护
- **中等**: 影响可读性，增加维护成本
- **轻微**: 代码风格问题，不影响功能

### 第三步：提出重构方案

针对每个问题，给出具体的重构方案：

1. **怎么改** - 具体操作步骤
2. **为什么这么改** - 好处是什么
3 **改完之后代码变成什么样** - 给出示例

### 第四步：评估风险

- 重构是否会影响现有功能？
- 需要做什么测试来确保不 regress？
- 改动范围有多大？

---

## 常见重构场景和处理方式

### 1. 超长组件/函数 > 500 行

**问题**: 一个组件做了太多事情，难以理解和维护

**解决方案**: 拆分

- 抽取可复用部分为独立组件
- 抽取自定义 Hook 处理逻辑
- 拆分状态管理到多个原子 Store

**示例**:

```
Before:
ProductList.tsx (800行) - 包含列表、筛选、分页、卡片渲染

After:
- components/ProductList/index.tsx (协调)
- components/ProductList/ProductCard.tsx (商品卡片)
- components/ProductList/FilterBar.tsx (筛选栏)
- hooks/useProductList.ts (逻辑抽取)
```

### 2. 重复代码

**问题**: 多个地方复制粘贴了相同逻辑

**解决方案**: 抽取复用

- 抽取为公共工具函数
- 抽取为自定义 Hook
- 抽取为公共组件
- 放到 `src/utils/` 或 `src/hooks/` 或 `src/components/`

### 3. 复杂条件分支

**问题**: `if-else` 嵌套太多，或者 `switch-case` 太长

**解决方案**:

- 使用对象映射替代条件分支
- 抽取分支逻辑为独立函数
- 使用策略模式

**示例**:

```typescript
// ❌ Before
function getStatusText(status: string) {
  if (status === 'pending') {
    return '待处理';
  } else if (status === 'paid') {
    return '已支付';
  } else if (status === 'shipped') {
    return '已发货';
  } else if (status === 'delivered') {
    return '已签收';
  } else {
    return '未知';
  }
}

// ✅ After
const statusMap: Record<string, string> = {
  pending: '待处理',
  paid: '已支付',
  shipped: '已发货',
  delivered: '已签收',
};

function getStatusText(status: string) {
  return statusMap[status] || '未知';
}
```

### 4. 命名不清晰

**问题**:

- `data`, `info`, `res` 这种太泛的命名
- 缩写没人看得懂
- 命名和实际做的事情不匹配

**解决方案**: 重命名为更具体的名字

- `data` → `productList`
- `info` → `userInfo`
- `res` → `response`
- 不要使用拼音命名

### 5. 类型问题

**问题**:

- 到处使用 `any`
- 重复定义相同类型
- 缺少类型定义

**解决方案**:

- 替换 `any` 为具体类型
- 将共用类型放到 `src/types/` 统一导出
- 为所有函数参数返回值添加类型

### 6. 不合理的导入路径

**问题**: 使用很长的相对路径 `../../../../`

**解决方案**: 使用路径别名 `@/`

```typescript
// ❌ Before
import { api } from '../../../../api/index.tsx';

// ✅ After
import { api } from '@/api/index.tsx';
```

### 7. 内存泄漏

**问题**:

- useEffect 添加了事件监听但没有清理
- setInterval/setTimeout 没有清除

**解决方案**: 添加清理函数

```typescript
// ❌ Before
useEffect(() => {
  window.addEventListener('resize', handleResize);
}, []);

// ✅ After
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, [handleResize]);
```

### 8. 过多的内联魔术数字

**问题**: 代码中直接使用魔法数字

**解决方案**: 抽取为命名常量

```typescript
// ❌ Before
if (count > 10) {
  showMore = true;
}

// ✅ After
const MAX_DISPLAY_ITEMS = 10;

if (count > MAX_DISPLAY_ITEMS) {
  showMore = true;
}
```

### 9. 违反单一职责

**问题**: 一个组件/函数做了多个不相关的事情

**解决方案**: 拆分

> "如果一个模块有两个以上改变它的理由，它就应该被拆分" --- 单一职责原则

示例：
- 不要让一个组件既处理表单又发送请求又跳转页面 → 拆分
- 不要让一个工具函数既格式化日期又请求API → 拆分

### 10. 过时的代码/依赖

**问题**: 使用了过时的 API 或第三方库

**解决方案**: 升级到现代用法

## 重构原则

### 男孩法则

> "每次你离开，都要让代码比你发现它的时候更干净"

不需要一次性全部重构，问题发现了就逐步修复。

### 三问重构

重构前问自己：

1. **我能理解现在这段代码做什么吗？** - 如果不能，需要重构
2. **修改这个功能容易吗？** - 如果不容易，需要重构
3. **这段代码会被经常修改吗？** - 如果会，需要重构

### 不要过度重构

- 简单问题简单处理，不需要设计模式过度设计
- YAGNI (You Ain't Gonna Need It) - 不要实现现在不需要的功能
- 能够工作并且易于理解就是好代码

---

## 优先级划分

### 🔴 必须立即改（高优先级）

- 内存泄漏
- 安全漏洞
- 类型错误（any 泛滥）
- 重复逻辑导致的bug风险

### 🟠 有空就改（中优先级）

- 拆分过长组件/函数
- 抽取重复代码
- 改善命名
- 清理死代码

### 🟡 可改可不改（低优先级）

- 代码风格
- 格式问题
- 不影响理解的命名
- 已经稳定很少修改的代码
