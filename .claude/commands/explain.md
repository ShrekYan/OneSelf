# 代码逻辑解释规范

当需要解释一段代码或一个模块时，遵循本规范输出**精简、结构化、有价值**的解释。本规范适配本项目（React 19 + TypeScript + MobX）技术栈。

---

## 核心原则

1. **由粗到细**：先说整体，再说细节
2. **重点突出**：复杂逻辑多讲，简单代码跳过
3. **不说废话**：不重复代码已经说清楚的东西
4. **回答为什么**：不仅说做了什么，更解释为什么这么做
5. **控制篇幅**：言简意赅，拒绝水文

**铁律**：
> 代码能看懂的不解释，只解释：
> - 复杂的算法逻辑
> - 微妙的设计决策
> - 容易踩坑的边界处理
> - 不直观的副作用

**篇幅控制**：
- 文件 < 100 行：不超过 **500 字**
- 文件 < 500 行：不超过 **1000 字**
- 文件 > 500 行：只解释核心模块，不逐函数铺陈

---

## 三步解释框架

### 第一步：概览 → 快速建立上下文
- **核心功能**：一句话概括解决什么问题
- **位置依赖**：在项目中位置，被谁使用，依赖了什么

### 第二步：详解 → 分析结构和关键逻辑
- **导出结构**：主要导出了什么，各自职责
- **关键逻辑**：只讲核心/复杂逻辑，代码自明的跳过

### 第三步：总结 → 提炼设计思路和注意事项
- **设计决策**：为什么这么设计，做了什么取舍
- **踩坑提示**：这里有什么容易误用的地方
- **关键点**：3 条以内核心总结

## 针对不同代码类型的解释重点

| 类型 | 解释重点 |
|------|----------|
| **React 组件** | Props 含义、内部状态、哪些 Store、副作用、重渲染时机。检查：是否使用 `useObserver`、CSS Modules 规范 |
| **自定义 Hook** | 封装了什么逻辑、返回什么、依赖、清理逻辑 |
| **MobX Store** | 管理什么状态、哪些 action、派生数据、使用位置。检查：`makeAutoObservable`、字段类型 |
| **工具函数** | 输入输出、边界处理、算法思路（如果复杂） |
| **API 模块** | 提供哪些接口、请求/响应类型、特殊配置（缓存/跳过错误提示）。检查：按业务拆分、统一导出 |
| **样式模块** | 布局结构、设计令牌、响应式规则。检查：kebab-case 命名 |
| **页面模块** | 业务流程、数据流向、用户操作、权限控制、页面参数 |

## 完整示例

## 📝 代码解释：`src/hooks/useCountDown.ts`

### 概览

**功能**: 实现一个可控的倒计时 Hook，支持开始/暂停/重置，常用于验证码按钮。

**位置**: `src/hooks/useCountDown.ts`
**被谁使用**: 验证码按钮、弹窗倒计时等需要倒计时的场景
**关键依赖**: React `useState`, `useEffect`, `useCallback`

### 详解

**主要导出**:
- `useCountDown` - 倒计时自定义 Hook

**关键逻辑**:

#### `useCountDown`
```typescript
export function useCountDown(initialSeconds: number) {
  // 状态定义 + 控制方法 + 定时器 effect
}
```
- **输入**: `initialSeconds: number` - 倒计时初始秒数
- **逻辑**: 用 `seconds` 存剩余时间，`isRunning` 存运行状态，提供三个控制方法。`useEffect` 监听依赖，每秒减 1，到 0 自动停止，清理函数清除定时器。
- **输出**: `{ seconds, isRunning, start, stop, reset }` - 当前状态和控制方法
- **边界处理**: 到 0 自动停止、依赖变化自动清理定时器、组件卸载清除定时器防止内存泄漏

### 总结

**设计决策**:
> 为什么把定时器放在 useEffect 里而不是 start 方法中？
>
> 因为依赖 `isRunning` 和 `seconds`，放在 useEffect 中 React 会自动管理生命周期，依赖变化时自动清理重建，代码简洁且不易漏清理。

**踩坑提示**:
- `setInterval` 有累积误差，对精度要求极高的场景不适用

**关键点**:
1. `setInterval` 实现每秒递减
2. useEffect 自动管理定时器生命周期
3. `useCallback` 保持方法引用稳定，避免不必要重渲染

## 需要避免

- ❌ 不要逐行翻译代码
- ❌ 不要说"这行定义了一个变量 x"这种废话
- ❌ 不要跳过复杂的设计决策
- ❌ 不要使用过于晦涩的黑话，用通俗语言解释
- ❌ 不要遗漏边界情况和错误处理的说明
- ❌ 不要隐瞒潜在问题，坦诚提示风险更有价值
