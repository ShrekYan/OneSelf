# 后端架构评估与优化建议报告

> 评估日期：2026-04-09
> 技术栈：NestJS 11 + Prisma 6.5 + TypeScript 5.7 + MySQL

## 1. 架构总结

### 整体评价

这是一个**设计良好、架构清晰**的小型博客系统后端 API 服务，采用了现代化的 NestJS 分层架构，遵循领域驱动设计原则，整体质量较高。

### 设计亮点

✅ **模块化拆分清晰** - 按业务领域（`auth`、`category`、`article`）正确拆分，公共基础设施统一放在 `common/`，符合 NestJS 最佳实践

✅ **完整的异常处理体系** - 三级异常捕获（`AllExceptionsFilter` + `BusinessExceptionFilter` + `PrismaExceptionFilter`），Prisma 错误码专门处理，统一 `ApiResult` 响应格式

✅ **已实现关键性能实践** - 并发查询优化（`Promise.all`）、事务保证数据一致性（点赞操作）、数据库索引设计合理、unique 约束正确使用

✅ **安全认证设计合理** - JWT 双令牌机制（access + refresh），refresh token 持久化支持吊销，全局 JWT 解析设计巧妙，密码 bcrypt 加密（cost factor 12）

✅ **可观测性基础良好** - 统一 JSON 格式请求日志，按级别分类便于采集分析

✅ **现代化技术栈** - NestJS 11 + Prisma 6.5 最新稳定版本，全面 TypeScript 类型安全，strict 模式开启，class-validator 自动校验

## 2. 性能优化建议

### 数据库查询优化

| 优化项 | 问题 | 建议方案 | 优先级 |
|--------|------|----------|--------|
| **全文搜索优化** | 当前搜索用 `contains` 导致全表扫描 | 添加 MySQL FULLTEXT 全文索引，使用原生全文搜索；数据量大时考虑 Elasticsearch/Meilisearch | ⭐⭐⭐ |
| **命中计数缓存** | 每次浏览都更新数据库，高并发写操作多 | 使用 Redis 缓存计数器，异步批量同步到数据库 | ⭐⭐ |
| **置顶文章缓存** | 置顶文章不常变化但每次都查库 | 使用 `@nestjs/cache-manager` 内存缓存，缓存 5-15 分钟 | ⭐⭐⭐ |
| **分页 COUNT(\*) 优化** | 每次分页都计算总数，大数据量全表扫 | 后续页使用 `hasMore` 模式，不需要总数 | ⭐ |
| **连接池配置优化** | Prisma 默认连接池 10，生产不足 | 调整 `connection_limit=20`，配置超时参数 | ⭐⭐⭐⭐ |

### 并发优化

- **点赞防雪崩**：高并发同一文章点赞时，使用 Redis 队列削峰，异步批量更新数据库，优先级 ⭐⭐

## 3. 安全性建议

### 认证授权

| 改进项 | 问题 | 方案 | 优先级 |
|--------|------|------|--------|
| **JWT 密钥强度校验** | 存在默认密钥 `default-secret` | 启动时强制检查 `JWT_SECRET` 必须存在且长度 ≥ 32，生产禁止默认值 | ⭐⭐⭐⭐⭐ |
| **刷新令牌轮换** | 刷新不换 refresh token，泄露风险 | 每次刷新生成新 refresh token， revoke 旧的 | ⭐⭐⭐ |
| **清理过期刷新令牌** | 过期令牌一直占用数据库 | 增加 cron 定时任务清理过期令牌 | ⭐⭐ |
| **登录失败锁定** | `handleLoginFailure` 预留未实现 | `Users` 表增加 `login_attempts` + `locked_until`，N 次失败锁定 15 分钟防暴力破解 | ⭐⭐⭐⭐ |
| **CORS origins 限制** | 当前允许所有来源 | 生产环境配置 `ALLOWED_ORIGINS` 白名单 | ⭐⭐⭐⭐ |

### 输入验证防护

| 改进项 | 问题 | 方案 | 优先级 |
|--------|------|------|--------|
| **限制分页最大 pageSize** | 无上限，可一次性拉全表 | DTO 添加 `@Max(100)`，限制最大 100 条/页 | ⭐⭐⭐⭐ |
| **关键词长度限制** | 无长度限制，长关键词导致全表扫 | 添加 `@Length(1, 50)` 限制 | ⭐⭐⭐ |

### 其他安全建议

| 改进项 | 方案 | 优先级 |
|--------|------|--------|
| **Rate Limiting 限流** | 使用 `@nestjs/throttler`，登录每分钟 5 次，API 每分钟 100 次 | ⭐⭐⭐⭐ |
| **Helmet 安全头** | 引入 `helmet` 设置各类安全响应头防 XSS/clickjacking | ⭐⭐⭐⭐ |

## 4. 可维护性优化

### 🔥 类型问题修复（最高优先级）

**问题**：部分 Prisma 模型使用全小写命名，导致 TypeScript 访问时需要 `as any` 绕过检查，类型不安全。

**修复方案**：
- 将 `schema.prisma` 中所有模型重命名为 **PascalCase**（保持 `@@map("table_name")` 不变）
- 例如：`model articles` → `model Articles`，`@@map("articles")` 表名映射不变
- 这样 Prisma 生成的类型正确，不需要 `as any` 绕过

**影响**：解决类型安全问题，避免潜在 bug。优先级 ⭐⭐⭐⭐⭐

### 代码组织改进

- **DTO 目录重构**：article 模块 DTO 按子目录过于细碎，建议同一模块 DTO 直接放 `dto/` 目录，优先级 ⭐⭐
- **扩展业务错误码**：补充限流、权限不足等错误码定义，优先级 ⭐⭐

### 测试覆盖率提升

当前有测试框架但缺少实际测试，建议优先补充：
- 认证相关 E2E 测试（注册、登录、刷新）
- 核心文章查询单元测试

优先级 ⭐⭐⭐
