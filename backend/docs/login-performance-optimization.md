# Login 接口性能分析与优化方案

## Context

当前 `login` 接口在 **100万用户数据 + 50 QPS 并发 + 50ms 网络延迟** 条件下，需要进行性能瓶颈分析并提供优化方案。

### 当前实现分析

**代码位置**: `backend/src/auth/auth.service.ts` - `login()` 方法 (line 193-242)

**当前流程**:
```
1. 根据 username 查询用户 → 一次 SQL 查询
2. bcrypt.compare 验证密码 → CPU 密集型计算 (cost = 12)
3. 生成 JWT (access + refresh) → 两次签名
4. saveRefreshToken 写入数据库 → 一次 SQL 插入
5. 返回结果
```

**数据库结构**:
- `Users.username` 已设置 `@unique` → 存在**唯一索引**，查询速度快 (O(log n))
- `RefreshTokens.refresh_token` 已设置 `@unique`

---

## 性能瓶颈分析

| 环节 | 是否存在瓶颈 | 原因 |
|------|-------------|------|
| **用户查询** | ❌ 无瓶颈 | username 有唯一索引，100万数据查询约 1-5ms |
| **bcrypt.compare** | ✅ **主要瓶颈** | bcrypt 是 CPU 密集型，cost 12 单次约 100-250ms。50 QPS 会占满 CPU |
| **JWT 签名** | ❌ 无瓶颈 | JWT 签名用 HMAC-SHA256，非常快，单次 < 1ms |
| **保存 refreshToken** | ⚠️ 次要瓶颈 | 每次登录都插入新记录，增加数据库写入压力 |

### 关键问题

**bcrypt cost = 12 的计算成本**:
- 单次 `bcrypt.compare` 约消耗 **150ms CPU 时间**
- Node.js 是单线程事件循环，**50 QPS ≈ 7.5 秒 CPU 负载** → CPU 会 100% 占满
- 响应延迟会急剧上升，请求排队等待

### 概念通俗解释（前端视角）

| 术语 | 解释 |
|------|------|
| **QPS** | Queries Per Second，**每秒请求数**。比如一秒钟 50 个人点登录，就是 50 QPS。 |
| **CPU 密集型** | 操作大部分时间都在占用 CPU 计算，不是在等网络/数据库。类似前端在 JS 主线程跑大循环，会卡住整个页面。Node.js 单线程，一个计算占 150ms，这 150ms 其他请求都得排队。 |
| **cost = 12** | bcrypt 故意设计得很慢来保证安全，`cost` 是计算成本因子。`cost = 12` 代表要重复计算 `2^12 = 4096` 轮。轮数越多越安全，但也越慢。 |

**为什么 50 QPS 会占满 CPU？**

算一笔账：
- 一次 `bcrypt.compare` ≈ 150ms = 0.15 秒
- 50 个请求同时来 → 总 CPU 需求 = `50 × 0.15s = 7.5s`
- 但 Node.js 单线程一秒只能干 1 秒的活 → **一秒钟需要 7.5 秒的计算量**

结果就是：
- CPU 直接 100% 占满
- 后面的请求都在排队等待
- 每个人的登录响应延迟都会蹭蹭往上涨

类比前端：相当于每次点击都要在主线程算一个 150ms 的循环，点 50 次，页面直接卡住不动了。

## 优化方案对比

提供三种优化方案，从易到难：

---

### 方案一：密码缓存优化 (最简方案)

**思路**: 对**最近登录成功**的用户，缓存密码哈希到 Redis，避免每次都查询数据库 + 不频繁做 bcrypt。

**流程变更**:
```
1. 先查 Redis 缓存 → 有缓存？
   ├─ 是 → 直接对比密码 (内存对比，无需 bcrypt！)
   └─ 否 → 查询数据库，走原有流程，写入缓存
2. 验证成功后生成 Token 并返回
```

**具体实现**:
- Redis 缓存结构: `cache:password:{username}` → 值为 `password_hash`
- 缓存过期时间: 5-15 分钟
- 只有**登录成功**才缓存（避免缓存不存在的用户名）
- 用户修改密码后主动删除缓存

**优缺点**:
| 优点 | 缺点 |
|------|------|
| 改动最小，风险低 | 仍有 bcrypt 计算（只是不频繁） |
| 大幅降低数据库负载 | 需要引入 Redis |
| 热点用户登录响应极快 | |

**性能提升**:
- **QPS 提升**: 对于热点用户，登录响应从 ~200ms → ~10ms
- **CPU 负载降低**: 减少 90% 以上 bcrypt 计算（多数命中缓存）
- **数据库压力降低**: 减少 80% 以上查询

---

### 方案二：异步化 + 分离计算 (中等方案)

**思路**: 将 bcrypt 计算从主流程中剥离，使用线程池/外部进程处理，避免阻塞 Node.js 事件循环。

**两种实现方式**:

**A. 使用 bcrypt 的并行执行 + 连接外部 bcrypt 服务**
- 启动独立的 bcrypt 计算 worker 进程池
- 通过 IPC 或网络将计算任务交给 worker
- Node.js 事件循环不被阻塞，可以继续处理其他请求

**B. 使用更快的密码算法** (推荐)
- 改用 **Argon2id** 算法，相较于 bcrypt 更快更安全
- 相同安全等级下，Argon2id 计算速度比 bcrypt 快 2-4 倍

**流程变更** (Argon2id 方案):
```
1. 查询用户 (不变)
2. 使用 argon2.verify 替换 bcrypt.compare
3. 后续流程不变
```

**优缺点**:
| 优点 | 缺点 |
|------|------|
| 算法层面提升速度，天然并发 | 需要迁移密码哈希 |
| 安全性更高 (Argon2 是密码哈希竞赛冠军) | 存量用户需要平滑迁移 |
| 不改变原有架构，改动较小 | |

**性能提升**:
- **计算速度提升**: 2-4x，单请求从 150ms → 40-70ms
- **CPU 负载降低**: 同样 QPS 下 CPU 使用率降低 60-70%
- **支持更高并发**: Node.js 不阻塞，可以处理更多请求

**平滑迁移方案**:
- 新用户直接用 Argon2
- 存量用户保留 bcrypt，登录成功后静默升级为 Argon2
- 代码中同时支持两种格式的哈希识别

---

### 方案三：完全分层架构 (彻底方案)

**思路**: 认证完全分离为独立服务，引入缓存层、前置快速拒绝，水平扩展能力。

**架构变更**:

```
┌─────────────┐
│  API 网关   │
└──────┬──────┘
       │
       ▼
┌─────────────┐  ┌──────────────┐
│ 认证服务     │  │ 用户缓存 Redis │
│ (无状态)     │←→│ (密码哈希缓存) │
└──────┬──────┘  └──────────────┘
       │
       ▼
┌─────────────┐
│  用户数据库  │
└─────────────┘
```

**核心优化点**:

1. **前置用户缓存**:
   - 热点用户信息（含密码哈希）全量缓存到 Redis
   - 90% 请求不用访问数据库

2. **登录失败计数与限流**:
   - 使用 Redis 记录 IP + 用户名的失败次数
   - 超过阈值直接拒绝，避免暴力破解同时保护数据库

3. **refreshToken 优化**:
   - refreshToken 也缓存到 Redis
   - 不用每次写入数据库，只在过期后清理

4. **水平扩展**:
   - 认证服务无状态，可以独立扩缩容
   - 应对突发登录流量（如促销活动）

**优缺点**:
| 优点 | 缺点 |
|------|------|
| 架构最优雅，支撑万级 QPS | 改动大，需要调整架构 |
| 完全隔离认证域，易于维护 | 运维复杂度增加 |
| 极限性能最好 | |

**性能提升**:
- **P99 延迟降低**: 从几百ms → 十几ms
- **支撑并发能力**: 50 QPS → **数千 QPS**
- **数据库压力**: 几乎卸下所有认证查询压力

---

## 方案对比表

| 指标 | 方案一 (密码缓存) | 方案二 (算法替换) | 方案三 (独立架构) |
|------|------------------|------------------|------------------|
| **实现难度** | ⭐ 简单 | ⭐⭐ 中等 | ⭐⭐⭐ 复杂 |
| **改动范围** | 小 (仅 AuthService) | 中 (密码模块) | 大 (架构重构) |
| **风险** | 低 | 中 (需迁移) | 中高 |
| **需要 Redis** | 是 | 可选 | 是 |
| **CPU 降低** | ~70-90% | ~60-75% | ~90-95% |
| **DB 压力降低** | ~80% | ~0% (不改变) | ~90%+ |
| **支撑 QPS** | 50 → ~200-300 | 50 → ~150-200 | 50 → **数千** |
| **推荐度** | ⭐⭐⭐⭐⭐ 推荐 | ⭐⭐⭐⭐ 推荐配合方案一 | ⭐⭐⭐ 大规模用 |

## 推荐实施建议

### 当前场景推荐：**方案一 + 方案二结合**

1. **第一步**：先实施**方案一**（密码缓存），改动最小，快速见效
   - 在 `AuthService` 中引入 Redis 缓存热门用户密码哈希
   - 解决 50 QPS 下的 CPU 瓶颈问题

2. **第二步**：再渐进实施**方案二**（算法迁移到 Argon2）
   - 新注册用户用 Argon2id
   - 存量用户登录成功后静默升级
   - 进一步降低 CPU 负载

### 修改的核心文件

| 文件 | 修改内容 |
|------|----------|
| `backend/src/auth/auth.service.ts` | 增加 Redis 缓存逻辑、修改密码验证流程 |
| `backend/src/auth/auth.module.ts` | 引入 Redis 模块 |
| `backend/prisma/schema.prisma` | Users 表增加 `password_algorithm` 字段（方案二需要） |
| `backend/src/config/` | 增加 Redis 配置 |
